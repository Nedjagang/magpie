# Audit query surface — design spec (v1.0)

**Status:** Design only. **No code yet.** Open questions in §13 must be confirmed (or working assumptions accepted) before implementation begins.

**Scope of this document:** the **Audit** sidebar entry — the queryable structured-event stream that's the operator's forensic view for "what changed and when." Covers filter bar, result rendering, payload detail surface, chain-verification status, cross-linking to host/scope/config/rollout drawers, deep-linking, performance budgets, retention. Does **not** cover the audit hash-chain Go layer (already shipped — see plan §6.1 #8 and `internal/audit/`), the per-surface embedded audit slices (host drawer's "Recent activity" — separate UI work that consumes the same backend), or audit shipping to external OTLP/SIEM (v1.x).

**Reading order:** §1–§3 frame why this surface matters and how it's structured; §4–§7 walk through filter + result + detail + cross-linking; §8 covers the chain verification status; §9–§11 cover state surfaces, deep-linking, performance + retention; §12–§16 cover accessibility, open questions, test surface, deferrals.

**Supersedes / complements:** [`docs/v1.0-plan.md`](./v1.0-plan.md), [`docs/v1.0-rollout-spec.md`](./v1.0-rollout-spec.md) §11 (event vocabulary), [`docs/v1.0-fleet-view-spec.md`](./v1.0-fleet-view-spec.md), [`docs/v1.0-host-drawer-spec.md`](./v1.0-host-drawer-spec.md) §8 (the host drawer's recent-activity slice consumes the same data, surfaced differently). On any conflict the plan wins for scope/cut-line questions; this spec wins for Audit-section UX.

## 1. Why this surface matters

Magpie v1.0 made every state change a tracked event (plan invariant 1). The hash-chain layer (plan §6.1 #8) made that record tamper-evident. This surface is what turns "we have the data" into "operators can find the answer." Three jobs at the same time:

1. **Forensic read.** "What happened to host `db-prod-7` in the last 24 hours?" → operator filters by `host_ref` + time, sees a chronological narrative ending at a `RolloutFailed` or `OverrideApplied` event.
2. **Cross-cutting search.** "Who's been pushing rollouts to `ship/linux` lately?" → operator filters by `scope_ref` + event type, sees the actors and their cadence.
3. **Tamper-evidence reassurance.** "Has the audit log been edited?" → chain verification status is one click away (or always visible) and answers in green/red.

If the operator reaches for SSH or grep against `audit_log` to answer any of these, the surface failed.

Two principles bind every decision in this spec:

- **Structured > free-text.** Each event renders from its typed fields (type / actor / scope / host / config / payload). The legacy v0.2 free-text Detail field is fall-back display only; the v1.0 vocabulary is what filters and sorts work against.
- **Cross-linkable everywhere.** Every entity reference in a result row (host_ref, scope_ref, config_ref, actor) is clickable and navigates to the relevant surface (host drawer, product drawer, config history, filtered audit view). Nothing is read-only ID text.

## 2. Where the operator comes from

Five entry paths, all converging on the same view:

1. **Sidebar click.** Default route — operator opens the tab, clicks "Audit" in the sidebar's third group (alongside Rollouts).
2. **Deep link.** A URL like `/audit?host=abc-123&since=24h` opens Audit pre-filtered. Filter state is URL-serialised so operators can paste a link in Slack: "the failure trail for db-prod-7 yesterday."
3. **Host drawer's "View all events for this host" link.** Opens Audit pre-filtered to that host's `host_ref`. The host drawer's embedded slice is intentionally tiny (5 most-recent events); the full history is here.
4. **Rollout drawer cross-link.** Each rollout's drawer carries a "View audit" link that pre-filters to events scoped to that rollout's `scope_ref` + `config_ref` (or filters by rollout id once per-rollout filtering lands — see §13 #5).
5. **Click on an actor in any other surface** (Fleet, Rollouts) → opens Audit filtered to that actor's events. Promotes "what has this operator been doing" from a separate workflow into a one-click drill.

All five preserve filter state on subsequent navigation within the session (§10).

## 3. Structural anatomy

The Audit view is a single column: header row + chain status + filter bar + result list + Load more.

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  Audit                          ✓ Chain intact (12,408 events)               │
├──────────────────────────────────────────────────────────────────────────────┤
│  Filter:  [Host  ▾ ] [Scope ▾ ] [Type  ▾ ] [Actor ▾ ] [Since 24h ▾ ]  Clear │
├──────────────────────────────────────────────────────────────────────────────┤
│  ● RolloutAborted  alice@aptean  ship/linux           12s ago  ▸             │
│    abort_reason=canary_gate_failed  applied=0/1  failed=1                    │
│  ● RolloutAdvanced system          ship/linux         2m ago   ▸             │
│    canary→soak                                                               │
│  ● RolloutCreated  alice@aptean   ship/linux  cfg #42  4m ago  ▸             │
│    phased canary_size=1 soak_seconds=300 gate_mode=auto                      │
│  ● OverrideApplied alice@aptean   instance: db-prod-7  3h ago  ▸             │
│    advertised: ship/linux → effective: pay/linux                             │
│  ...                                                                         │
│                                                                              │
│  Showing 50 of recent                                  [ Load more ▸ ]       │
└──────────────────────────────────────────────────────────────────────────────┘
```

Four structural zones:

- **Header row.** Page title + chain-verification status pill (§8) — always visible.
- **Filter bar.** Five filter chips: host / scope / type / actor / time-range, plus a "Clear" link. (Six if `scope_kind` becomes a separate filter — see §4.)
- **Result list.** Reverse-chronological event rows. Click a row to expand its full payload inline (§6).
- **Load more.** Cursor-paginated, "Showing N of recent" plus a button that appends the next page. No infinite scroll (matches Fleet view spec rationale: easier ARIA, clear stopping point).

## 4. Filter bar

Five filter chips, always visible, click each to open a small dropdown / autocomplete:

### 4.1 Host

Single-select. Free-text input with autocomplete from currently-known agent uids (queries `/api/v1/agents` with a search prefix). Setting this filter limits results to events where `host_ref` matches.

The chip label shows the host's friendly name (`host.name` attribute) if known, the truncated uid otherwise.

### 4.2 Scope

Two-stage: pick scope_kind first (`product_variant` / `instance`), then narrow within it. For `product_variant`, the second stage is the same product+variant picker the Fleet view uses (autocomplete from cohorts present in the fleet). For `instance`, it's the same host picker as §4.1 (instance_uid is the scope_ref for instance scope).

### 4.3 Type

Multi-select dropdown of the structured event types from spec §11:

- Rollout: RolloutCreated / RolloutInstant / RolloutAdvanced / RolloutPaused / RolloutAborted / RolloutPromoted / RolloutFastPromoted
- Override: OverrideApplied / OverrideCleared
- Host: LabelChanged / HostDeleted
- Auth: AuthSucceeded / AuthFailed *(when those land)*
- Legacy: a single bucket "v0.2 free-text" that captures rows with `type = ''` (the pre-v1.0 events that don't have a structured type — surfaced for completeness, hidden behind an opt-in toggle).

Multi-select because operators commonly want "any rollout event" or "any override event" — being able to tick a category in one click matches the natural query shape.

### 4.4 Actor

Free-text autocomplete drawn from distinct actors in recent events. The "system" pseudo-actor (used for engine-driven transitions like AdvancePhase) is included.

### 4.5 Since

Single-select dropdown:

- 1 hour
- 24 hours (default)
- 7 days
- 30 days
- All time *(disabled when result set would exceed 50k rows; see §11 for the threshold)*

### 4.6 Clear

Resets every filter chip. URL state cleans up to defaults. (Does not reset chain-verification status.)

## 5. Result row anatomy

Each row is a compact two-line pattern that fits dense scanning while keeping cross-link affordances visible.

### 5.1 Top line

```
●  [TypePill]   actor   →   refs   ·   N ago   ▸
```

- **Dot** — colour-coded per event class. Rollout events = neutral; Aborted = red; OverrideApplied/Cleared = yellow; HostDeleted = red; system events = muted dot.
- **Type pill** — the structured event type, in code style. For `v0.2 free-text` rows the pill reads `legacy` and the `action` field renders next to it.
- **Actor** — clickable; clicking adds an actor filter (§4.4).
- **→ refs** — entity references shown contextually:
  - Rollout events: `scope_ref` + optional `config_ref` (e.g. `ship/linux · cfg #42`)
  - Override events: `instance: db-prod-7` or `host db-prod-7 → ship/linux`
  - Host events: `host db-prod-7`
  - Each ref is a clickable cross-link (§7).
- **N ago** — relative time. Absolute on hover.
- **▸** — affordance for "click to expand" (§6).

### 5.2 Bottom line

A single-sentence summary of the event's payload, type-specific. Examples:

| Type | Bottom line |
|---|---|
| RolloutCreated | `phased canary_size=1 soak_seconds=300 gate_mode=auto` |
| RolloutInstant | `instant scope=ship/linux` |
| RolloutAdvanced | `canary→soak` (or whatever transition) |
| RolloutPaused | `paused from soak` |
| RolloutAborted | `abort_reason=canary_gate_failed applied=0/1 failed=1` |
| RolloutPromoted | `manual mode promote` |
| RolloutFastPromoted | `gate_status_at_skip=passed` |
| OverrideApplied | `advertised: ship/linux → effective: pay/linux` |
| OverrideCleared | `was: pay/linux` |
| LabelChanged | `from: ship/linux  to: ship/windows` |
| HostDeleted | `last_seen 8d ago at deletion` |

The summary is rendered from the structured `payload_json` fields, not from the legacy free-text Detail. v0.2 legacy rows fall back to the Detail string verbatim.

### 5.3 Visual density

Default: comfortable (~50px row height, two clearly separated lines). A density toggle at the top of the list switches to compact (one-line, payload summary trimmed) for operators who want to see more rows on screen — same pattern as Fleet view's density toggle. Per-session, defaults to comfortable.

## 6. Detail expansion

Clicking a row's `▸` (or anywhere on the row outside an entity link) expands an inline detail panel below it. Click again to collapse.

The panel shows:

```
┌─ #r1247  RolloutAborted  alice@aptean  4m ago ────────────────────────────┐
│                                                                            │
│  Type           RolloutAborted                                             │
│  Actor          alice@aptean                                               │
│  Scope          product_variant: ship/linux                                │
│  Config         #42                                                        │
│  Host           —                                                          │
│                                                                            │
│  Payload                                                                   │
│  {                                                                         │
│    "rollout_id": 7,                                                        │
│    "state": "aborted",                                                     │
│    "abort_reason": "canary_gate_failed",                                   │
│    "applied": 0, "failed": 1, "total_canary": 1                            │
│  }                                                                         │
│                                                                            │
│  Hash chain                                                                │
│  prev:  3a91…f02e        position: 12,407 of 12,408                        │
│  this:  7c02…ab94        ✓ verified by VerifyChain at 12s ago              │
│                                                                            │
│  [ View rollout #7 ▸ ]   [ View on host filter ▸ ]   [ Copy event JSON ]   │
└────────────────────────────────────────────────────────────────────────────┘
```

Top-half is the structured fields; payload renders pretty-printed JSON (collapsed at depth 2 for nested structures, expandable). Hash chain section shows the prev/this hash short forms, the position in the chain, and the chain-verification status as of last `/verify` call.

Bottom-row buttons:

- **View rollout #N** — navigates to the Rollouts drawer for that rollout. Visible only when `payload_json.rollout_id` is set.
- **View on host filter** — re-filters Audit to host_ref for that event's host (when host_ref is set). Useful when you arrived from a scope-filtered view and want to drill into the specific host.
- **Copy event JSON** — copies the full event (top-half + payload) as JSON to clipboard for sharing in Slack / pasting into a ticket.

### 6.1 Why inline expand and not a drawer

The row's structured fields are short and the payload fits in ~6 lines for any v1.0 event type. A right-side drawer would be heavyweight and hide context (the rows above and below). Inline expansion lets the operator scan a result list with one row open at a time as a "current focus," scroll up/down to see nearby events, and re-open the next row in the same flow.

## 7. Cross-linking

Every entity reference is clickable and navigates to the most useful surface for that ref:

| Ref | Click behavior |
|---|---|
| `host_ref` | Opens the host drawer for that instance_uid (page-level state set; existing UI). If the host has been deleted, the drawer opens in the spec'd "deleted host" state from host drawer §12. |
| `scope_ref` (product_variant) | Selects the product in the sidebar; opens the product drawer at the relevant variant. |
| `scope_ref` (instance) | Same as `host_ref` — instance scope_ref *is* the instance_uid. |
| `config_ref` | Opens the product drawer at that config's variant, History tab, scrolled to that revision. |
| `actor` | Re-filters Audit to events where actor matches. |
| `rollout_id` (in payload) | Opens the Rollouts drawer for that rollout. Surfaced in the detail panel's button row, not the result row, since rollout_id is a payload field rather than a top-level column. |

Cross-link clicks update the URL state (filter or selection) so the operator can back-button their way through their drilling.

## 8. Chain verification status

A pill in the Audit page header that shows the most recent `GET /api/v1/audit/verify` result. Always visible.

States:

```
✓ Chain intact (12,408 events)              ← green pill
↻ Verifying…                                ← spinner during fetch
✗ Chain broken at #1402, valid up to #1401  ← red pill, clickable
```

- **Intact** — no UI affordance; pill is informational.
- **Verifying** — short-lived during the periodic re-verify (every 60s while the Audit page is open, or on demand via a refresh button).
- **Broken** — clicking the pill scrolls + opens the detail panel for the row at `broken_at_id`. The detail panel's hash-chain section calls out the mismatch ("expected prev: X, found prev: Y"). Operator's clear next step: investigate, then either restore from off-host audit shipping (planned, v1.x) or accept the gap and continue from the next intact row.

The pill is also persistent across navigation within the session — switching to Fleet/Rollouts/Products and coming back doesn't re-verify; the cached status renders immediately, then refreshes in the background.

### 8.1 Why a pill, not a banner

The chain-intact case is by far the common one; an always-visible green banner is theatre. A pill that goes red on a real finding is the right "default-quiet, loud-when-needed" balance. Operators investigating a specific finding click into it; everyone else gets the reassurance without screen real-estate cost.

## 9. Empty / loading / error states

### 9.1 Empty system

Brand-new install, no events yet:

```
Audit
✓ Chain intact (0 events)

No audit events recorded yet. Events appear here as operators
publish configs, re-assign hosts, and rollouts complete.
```

### 9.2 Empty filter

Filter set returns zero matches:

```
No events match the current filter.

[ Clear filters ]
```

The Clear button is the same as the filter bar's Clear (§4.6).

### 9.3 Loading

Skeleton rows (8 lines) while the first page fetches. After the first page lands, "Load more" shows a brief inline spinner (no full-screen flash).

### 9.4 API error

Generic banner-error with retry:

```
⚠ Couldn't reach magpied: <reason>

[ Retry ]
```

The chain-verification pill goes amber with `?` if `/verify` fails — same retry semantic.

### 9.5 Auth expired (401)

Falls into the existing global sign-in modal (no Audit-specific handling).

## 10. Deep-linking / URL state

Filter state is URL-serialised. Default values are omitted to keep links short:

| Param | Values |
|---|---|
| `host` | instance_uid |
| `scope` | "kind:ref" e.g. `product_variant:ship/linux` or `instance:db-prod-7` |
| `type` | comma-separated EventType names; multi-select |
| `actor` | string |
| `since` | `1h | 24h | 7d | 30d | all` |

URL state syncs both ways: chip change updates URL, URL change (back/forward, manual edit, paste) updates chips.

Examples:

- `/audit?host=db-prod-7&since=24h` — events affecting db-prod-7 in last 24h.
- `/audit?type=RolloutAborted,RolloutFailed` — *(RolloutFailed isn't a real type today; placeholder for "any rollout failure")* — every failed rollout in the default 24h window.
- `/audit?scope=instance:db-prod-7` — events for the instance scope of db-prod-7 (override applies / clears, instance-Rollouts).

Cursor pagination is **not** in the URL — same rationale as Fleet view spec.

## 11. Performance + retention

### 11.1 Pagination

`/api/v1/audit?cursor=…&limit=…` is already cursor-paginated (default 100 / max 5000). The Audit UI uses default 100 per page; operators load more for deeper drills.

### 11.2 Filter performance

Backend filter support beyond `cursor` and `limit` doesn't ship today (the audit endpoint takes pagination only). v1.0 audit-section UI initially **filters server-side via the existing query params + client-side narrowing** within the loaded page — same model as Fleet view's first slice.

For server-side filter parity at scale, plan §6.1 + this spec §13 #4 propose adding `?host_ref=…&scope_kind=…&scope_ref=…&type=…&actor=…&since=…` query params to `/api/v1/audit`. That's a small handler change + index assertion (the `idx_audit_host`, `idx_audit_scope_v1`, `idx_audit_type` indexes from migration 00009 cover the cases). Working assumption: the server-side filter lands in the same code-track turn as the audit-section UI.

### 11.3 Render budget at scale

- **Cold first paint** with 100 events: < 1.5 s.
- **Load more** appends in < 300 ms at 100-row pages.
- **Filter chip change** that requires a server round-trip: < 500 ms p95.
- **Detail expand** (no fetch — uses already-loaded payload_json): instant.

Rendering uses windowing within the loaded set when result count exceeds 500. Below that, plain DOM.

### 11.4 Retention

Per decisions doc: indefinite for v1.0. Operators with finite-storage concerns surface them via dashboards (the `audit_log` table size isn't currently exported as a metric — adding `magpie_audit_log_rows` is plan §6.1 #9 territory; flagged in §13 #6 here).

The header pill shows total event count: "✓ Chain intact (12,408 events)". Operators eyeballing this number can spot runaway growth without instrumentation.

### 11.5 Polling cadence

Audit list is **not** auto-polled. Audit is forensic, not live — auto-refresh would shuffle rows the operator was reading. A "Refresh" button at the top right re-fetches on demand, and switching filter chips refetches naturally. Chain-verify pill *does* poll, every 60 s, in the background.

## 12. Accessibility

- Filter chip dropdowns are keyboard-navigable; arrow keys move focus, Enter selects.
- Result rows are real `<button>` elements (or `<li>` with `role="button"`) so Enter expands.
- Expanded detail panels announce via `aria-live="polite"` on first expand so screen-reader users hear the content land.
- Hash chain status pill has visible text and `aria-label` describing the state ("Chain intact, 12,408 events").
- All entity refs are real anchors / buttons with discrete roles.
- Type pill colour is *not* the only signal — every event class has a glyph or text affordance.
- Time strings are relative on screen, absolute via `<time datetime="…">` for screen readers reading absolute timestamps.

| # | Question | Working assumption |

## 14. Test surface

### Component tests

- Filter chips render and toggle correctly.
- URL state syncs both directions (chip change → URL; URL change → chip).
- Result row renders type pill + actor + refs + relative time + summary.
- Click on actor adds actor filter.
- Click on host_ref opens host drawer (or its deleted-host fallback).
- Click on scope_ref navigates to the right product / host drawer.
- Detail expansion toggles inline; payload renders pretty-printed.
- Chain pill renders intact / broken / verifying states.

### Integration tests (with magpied)

- First page loads from `/api/v1/audit`.
- Filter chip change triggers refetch with the right query params.
- "Load more" follows `X-Next-Cursor` header.
- `GET /audit/verify` drives the chain pill.
- 401 → global sign-in modal.

### Failure-mode tests

- API 5xx → banner-error with retry.
- Chain broken → pill goes red, click scrolls to broken row.
- Empty filter result → empty-filter copy renders.
- Auth expiry mid-page → sign-in modal; filter state preserved across re-auth.

### Accessibility tests

- Tab order: Refresh → chain pill → host chip → scope chip → type chip → actor chip → since chip → clear → first row.
- Detail panel announcement on expand.
- Screen reader announces chain-status changes via aria-live.

## 15. Out of scope for this spec

- **Hash-chain Go layer** — already shipped (plan §6.1 #8).
- **Per-surface embedded audit slices** — host drawer §8 (Recent activity), rollout drawer's audit cross-link. They consume the same backend; their UI is part of those specs.
- **Audit shipping to external OTLP / SIEM / S3** — operationally important at scale; v1.x.
- **Off-host audit replication** for tamper-evidence beyond the chain (e.g. signed daily snapshots) — v1.x.
- **Audit retention policies** — v1.x once growth observability lands (§13 #6).
- **Per-rollout audit view at the Rollouts drawer** — separate increment; for v1.0 the drawer cross-links to a scope+config-filtered Audit page.
- **Full-text search across `detail` / `payload_json`** — v1.x; structured filtering covers the v1.0 needs.
- **Audit export to CSV / JSON file from the UI** — the "Copy event JSON" button covers single events; bulk export is v1.x.
