# Rollout primitive — design spec (v1.0)

**Status:** Design only. **No code yet.** Open questions in §13 must be answered (or working assumptions confirmed) before implementation begins.

**Scope of this document:** the Rollout primitive — its lifecycle, state machine, gates, operator controls, API surface, schema, and audit semantics. This is the keystone primitive of Magpie v1.0; the publish dialog, Rollouts UI section, ApplyState surfacing, and audit views all derive from it.

**Reading order:** §1–§4 establish the model; §5–§8 are the operational rules; §9–§11 are the API/schema/audit; §12 covers failure handling; §13 calls out unanswered decisions; §14 covers test surface.

**Supersedes / complements:** the v1.0 plan at [`docs/v1.0-plan.md`](./v1.0-plan.md). On any conflict, the plan wins for scope/cut-line questions; this spec wins for Rollout-specific mechanics.

---

## 1. What a Rollout is

A Rollout is **the unit of intentional state change in Magpie**. Every transition from one live Config to another for a given Scope must be effected through a Rollout. There are no implicit broadcasts.

A Rollout binds together:

- A target **Scope** (`product+variant` or `instance`).
- A target **Config** (the YAML being rolled out).
- A **prior Config** (what was live at Rollout creation time — the rollback target).
- A **kind**: Phased (default) or Instant.
- **Phasing parameters**: canary size, soak window, gate mode.
- **Lifecycle state** + timestamps for every transition.

Rollouts produce structured **AuditEvents** at every state transition and operator action.

The "live" Config of a Scope is derived state: the Config of the most recent Rollout in `done` state for that Scope. There is no separate "scope_live_config" table — querying rollouts answers it directly, and the index supports it at fleet scale.

---

## 2. Rollout kinds

### Phased (default)

`validating → canary → soak → promoting → done`. The only safe shape at 2000-host scale. Default for any Publish action.

### Instant

`validating → promoting → done`. Skips Canary and Soak.

**Constraints on Instant:**

- Requires explicit toggle in the publish dialog. Default is unchecked.
- Audit-typed distinctly: `RolloutInstant` rather than `RolloutCreated`.
- Same Validate gate runs (semantic + structural). A semantically invalid config is rejected even in Instant mode.
- Visible in the Rollouts UI section with a distinct badge ("Instant" pill in amber) so post-incident review can find them.
- The publish dialog, when Instant is selected, replaces the Phased preview with: **"Instant Rollout — bypasses Canary and Soak. All N hosts receive the config in ~2s. Use only for emergencies."** The toggle itself is amber-colored.

**Use cases:** emergency rollback (Phased rollout aborted, then Instant on the prior Config to converge); hotfix where soak is unaffordable; brand-new Scope onboarding where there's no prior Config to canary against.

---

## 3. State machine

### States

| State | Meaning | Terminal? |
|---|---|---|
| `validating` | Validate phase: semantic + structural check running | No |
| `canary` | Canary phase: config is being delivered to canary subset | No |
| `soak` | Soak phase: watching ApplyState, gate evaluation in progress | No |
| `promoting` | Promote phase: config being delivered to remaining hosts | No |
| `paused` | Operator paused; advancement halted | No |
| `done` | Rollout complete; target Config is now live for the Scope | **Yes** |
| `aborted` | Rollout aborted; live Config remains the prior Config | **Yes** |

### Transitions

```
[create]                                      → validating

validating  validate pass + Phased            → canary
validating  validate pass + Instant           → promoting
validating  validate fail                     → aborted (no host saw config)

canary      all canary apply_state ≠ pending  → soak

soak        gate auto pass + auto mode        → promoting
soak        gate auto pass + manual mode      → soak (gate_passed_at set, await operator)
soak        gate auto fail                    → aborted (always; see §6)
soak        operator promote (manual mode)    → promoting (only if gate_passed_at set)
soak        operator pause                    → paused (prev_state=soak)
soak        operator abort                    → aborted
soak        operator fast-promote             → promoting

promoting   all apply_state ≠ pending         → done
promoting   operator pause                    → paused (prev_state=promoting)
promoting   operator abort                    → aborted

paused      operator resume                   → soak | promoting (from prev_state)
paused      operator abort                    → aborted
paused      operator fast-promote             → promoting (only if prev_state=soak)
```

### Pause semantics

- Pause halts advancement only. In-flight pushes complete; ApplyState updates continue.
- The Soak window clock pauses with the state. Resume re-evaluates the gate immediately (because conditions may have changed during pause — apply state could have shifted, hosts could have gone unhealthy).
- Pause is allowed in `soak` and `promoting`. **Not allowed in `validating`** (Validate is fast and atomic) or `canary` (Canary is "deliver to N hosts" which is a fan-out the system runs to completion before evaluating; pausing mid-fan-out is a complexity v1.0 doesn't need).

### Concurrency rule

**At most one non-terminal Rollout per Scope at any time.** Non-terminal = state ∉ {`done`, `aborted`}.

A Publish to a Scope with an in-flight Rollout returns `409 Conflict` with the in-flight Rollout's id. The operator chooses to wait, abort the in-flight, or coordinate. **Auto-abort or queueing is explicitly not v1.0 behavior** — too easy to lose work or create surprise.

## 4. Phases in detail

### 4.1 Validate

- **What happens:** the Config's YAML is run through (a) Magpie's structural checks (allowlist enforcement, pipelines reference defined receivers/exporters, etc.) and (b) the OTel collector's own config loader (`otelcol-contrib --config <yaml> --dry-run` equivalent, or in-process loader call).
- **Inputs:** the Config row, magpied's allowlist env vars, the bundled `otelcol-contrib` binary's config loader.
- **Outputs:** validation_status set on the Config row. Transition to `canary` (Phased) or `promoting` (Instant) on pass; transition to `aborted` on fail with `abort_reason` describing the validation error.
- **Duration budget:** < 5 seconds. If validation hangs past 5s, magpied times out and aborts the Rollout with `abort_reason=validate_timeout`.
- **Side effect:** zero hosts see this config. A failed-validate Rollout is purely local to magpied.

### 4.2 Canary

- **What happens:** magpied selects the canary subset and pushes the new Config to those hosts via OpAMP.
- **Subset size:** the larger of `(canary_pct% of currently-connected hosts in Scope, rounded up)` and `canary_count`. Defaults: see §13. Both parameters are optional in the request; magpied applies defaults.
- **Subset selection:** stable hash of `instance_uid` mod canary size — same hosts canary repeatedly across Rollouts rather than random churn. Operators can predict who's affected, which matters at 2000-host scale where you don't want to "guess who's a canary today."
- **Edge case — fewer connected hosts than canary target:** use what's connected. Gate evaluates against the actual canary size. **If zero connected hosts in Scope at canary-start, the Rollout aborts with `abort_reason=no_canary_targets`** (working assumption — see §13 #6).
- **ApplyState:** rows created with `is_canary=true`, `state=pending`. Transitions to `applying` when the OpAMP message is sent. Transitions to `applied` or `failed` based on agent heartbeat report.
- **Outputs:** transition to `soak` once all canary `apply_state` rows have left `pending` state — every canary host has either applied, failed, or remained disconnected long enough to time out (30s push ack window).

### 4.3 Soak

- **What happens:** wait window watching ApplyState updates from canary hosts. Default window: 5 minutes.
- **Gate evaluation (continuous):** every 10 seconds during Soak, magpied evaluates:
  1. **Apply success rate.** Fraction of canary `apply_state` rows in `applied` state (out of total canary size). Threshold: ≥ 95%. Below threshold → gate fails immediately.
  2. **Health regression.** Any canary host that was healthy at Rollout creation and is now unhealthy → gate fails immediately.
  3. **Pending ApplyState at window end.** A host in `pending` at window end (couldn't be reached) counts as "not applied" for rate calculation. Working assumption — see §13 #4.
- **Gate pass:** all conditions hold continuously for the full Soak window.
- **Gate fail:** auto-abort fires immediately on first gate failure (§6).
- **Outputs:**
  - Auto mode: gate pass → transition to `promoting`.
  - Manual mode: gate pass → stay in `soak` with `gate_passed_at` timestamp set. Operator must hit Promote to advance.
- **Manual mode rationale:** for change-management environments where every promotion needs operator sign-off even when the canary is healthy. Auto mode is the default and what most v1.0 operators will use.

### 4.4 Promote

- **What happens:** magpied pushes the Config to all hosts in Scope not already in canary.
- **Membership at promote-start:** dynamic. Currently-connected hosts in the Scope, minus the canary set, are targeted. Hosts that joined the Scope after canary-start are included.
- **ApplyState:** rows created with `is_canary=false`, lifecycle same as canary.
- **Outputs:** transition to `done` once all `apply_state` rows for the rollout have left `pending`. Failures during Promote do not automatically abort — they're reported as `failed` ApplyState but the Rollout still reaches `done`. (See §13 #5 for whether this should change.)
- **Why no Promote-failure auto-abort:** at promote time, the Config has already passed canary + soak, so per-host failures at promote are most often "this specific host has a local problem" rather than "the config is bad." Auto-aborting on a single Promote failure punishes the entire Scope for one bad host. Surface failures clearly; let humans decide.

### 4.5 Done

- Terminal. The Rollout's Config is now the live Config for the Scope.
- Live Config resolution at any time: per-Scope, the most recent Rollout in `done` state determines the live Config.
- Hosts that join the Scope after Done get the live Config on connect (no further pushing needed).

## 5. Operator controls

All operator actions are bearer-token-authenticated and produce AuditEvents.

### Pause

- **Where:** `soak`, `promoting`.
- **Effect:** halts gate evaluation (Soak) or push fan-out (Promote). In-flight pushes complete normally. `prev_state` recorded so Resume knows where to return.
- **Audit:** `RolloutPaused` with `paused_from_state`.

### Resume

- **Where:** `paused`.
- **Effect:** transition back to `prev_state`. Re-evaluate gate immediately on resume from `soak` (conditions may have changed during pause). For resume from `promoting`, just continue fan-out.
- **Audit:** `RolloutAdvanced` with `from_state=paused`, `to_state=<prev_state>`.

### Abort

- **Where:** any non-terminal state.
- **Effect:**
  - Rollout transitions to `aborted` with `abort_reason` and `aborted_at`.
  - Hosts that received the new Config during canary or promote get a re-push of the prior Config (the Scope's still-live Config).
  - The re-push is **not** a new Rollout — it's a recovery action, since the Scope's live Config didn't change (Rollout never reached Done). This avoids audit-log noise of "rollback rollout to recover from aborted rollout."
  - ApplyState rows are preserved for audit / forensics.
- **Audit:** `RolloutAborted` with `abort_reason` ∈ {`operator`, `canary_gate_failed`, `validate_failed`, `validate_timeout`, `no_canary_targets`} and `hosts_rolled_back` count.

### Fast-Promote

- **Where:** `soak`, or `paused` (when `prev_state=soak`).
- **Effect:** skip remaining gate evaluation, immediately transition to `promoting`. Validate already passed; this only bypasses the Canary→Soak gate.
- **Constraint:** *not* available before Canary is complete (i.e. not in `validating`, not in `canary`).
- **Audit:** `RolloutFastPromoted` with `gate_status_at_skip` (whether gate had passed, failed, or was still evaluating).
- **UI:** the action button shows an amber warning explaining gates are being skipped, with the current ApplyState aggregates shown so the operator sees what they're committing to. Confirm dialog enumerates: "Skip remaining Soak window. Proceed to Promote? Canary apply rate is currently X/Y. This action is audit-logged."

### Promote (manual mode only)

- **Where:** `soak` with `gate_passed_at` set.
- **Effect:** transition to `promoting`. Identical to auto-mode auto-promote, just operator-triggered.
- **Audit:** `RolloutPromoted`.

## 6. Auto-rollback on canary gate failure

**Working assumption (§13 #3, also flagged in plan §8):** auto-abort on canary gate failure is **default-on** in v1.0.

Reasoning:

- Canary failure means a config has reached ≤ ~5% (or N=10) of the fleet and is failing. The blast radius is bounded by the canary size — failure is contained.
- Leaving a failing rollout in `soak` for human review just gives the failure more time to compound (more hosts retry-and-fail, agent restart loops, etc.).
- Auto-abort triggers Abort semantics (§5): rollback push to canary hosts; prior Config remains live.
- Audit trail clearly shows `RolloutAborted` with `abort_reason=canary_gate_failed` so the operator sees why immediately.

**Override path:** an operator who specifically wants to inspect a failing canary before it auto-aborts can hit Pause as soon as gate evaluation reports failed (the Rollouts UI surfaces gate state in real-time, updated every 10s). Once paused, they can Abort manually or Resume (gate re-evaluates and may auto-abort again). Pause must be possible *before* the auto-abort fires — implementation must give the operator at least one polling cycle (default 10s) of warning state before tripping abort. Concretely: gate evaluation result is published to UI on every cycle; auto-abort fires on the cycle *after* failure is detected, not the same cycle.

This decision should be revisited if real-world operations show it to be wrong.

## 7. Resolution priority and per-host overrides

A host's effective Config at any moment, in resolution order:

1. **Instance Scope:** if the host's `instance_uid` has a Rollout in `done` state for an `instance` Scope → that Rollout's Config.
2. **Product+Variant Scope:** else if the host's effective `(product, variant)` has a Rollout in `done` state → that Rollout's Config.
3. **Default:** else the `(product=default, variant=default)` Scope's live Config.
4. **Last-known-good:** else the agent's local cache (control plane outage scenario; preserved from v0.2).

### Implications for in-flight Rollouts

- **A host with an instance override does NOT receive product+variant Rollouts.** Those Rollouts skip the host entirely — the host doesn't appear in canary candidates or promote targets for them.
- **An instance Rollout for that host targets only that host.** Canary set is just the host; Soak still applies, but the gate is trivially "did this one host apply successfully." The full state machine still runs — there's no special "single-host" path.
- **Clearing an instance override** (deleting the override Rollout / running a new instance Rollout that supersedes it) does not magically re-trigger product+variant Rollouts. The host's resolved Config flips on next resolution; the agent re-applies on next push (heartbeat or explicit re-push). The state of in-flight product+variant Rollouts is unaffected.
- **A host joins the Scope mid-Rollout:** if it joins before Promote starts, it's included in Promote (but not retroactively in Canary). If it joins after Done, it gets the now-live Config on next push.

## 8. Apply timing and OpAMP push semantics

ApplyState transitions tied to OpAMP message lifecycle:

- `pending` → row created when the rollout enters Canary (for canary hosts) or Promote (for promote hosts).
- `pending` → `applying` when magpied sends the `ServerToAgent` message containing the new Config.
- `applying` → `applied` when the agent reports `effective_config_hash` matching the rollout's target Config hash in its next OpAMP `AgentToServer` heartbeat.
- `applying` → `failed` when the agent reports `RemoteConfigStatus.Failed` with an error message.
- Stuck in `applying` for > 60s → considered a soft failure for the purposes of gate evaluation (counts as not-applied), but the row stays in `applying` (not flipped to `failed`) so a delayed report can still complete it.

**Retries:**

- v1.0 does not retry pushes from magpied's side. The agent retains the last config push and applies on its own; if it fails to apply, that's reported and stays reported.
- Operator retry path: abort the Rollout, fix the config, create a new one. This is intentional — auto-retry of a failing config across the fleet is exactly the "config typo cascades to outage" scenario v0.2's instant-broadcast had.
- Per-host attempt_count exists in the schema for future use; v1.0 it stays at 1.

## 9. API surface

All routes under `/api/v1`. Bearer token required (carried over from v0.2).

### Create a Rollout

```http
POST /api/v1/rollouts
Authorization: Bearer <token>
Content-Type: application/json

{
  "scope_kind":   "product_variant" | "instance",
  "scope_ref":    "ship/linux" | "<instance_uid>",
  "config":       { "name": "...", "yaml": "...", "tags": ["..."] },
  "rollout_kind": "phased" | "instant",
  "canary":       { "pct": 5, "count": 10 },        // optional; defaults from §13
  "soak_seconds": 300,                              // optional; default 300
  "gate_mode":    "auto" | "manual"                 // optional; default "auto"
}
```

Responses:

- `201 Created` → `{ "rollout_id": 42, "state": "validating", ... }`
- `400 Bad Request` → `{ "error": "validate_failed", "details": "<error from collector loader>" }`
- `409 Conflict` → `{ "error": "rollout_in_progress", "existing_rollout_id": 41 }`
- `403 Forbidden` → `{ "error": "scope_disallowed_components", "details": "..." }` (allowlist hit)

### Read a Rollout

```http
GET /api/v1/rollouts/:id
```

Response includes:

- All Rollout fields (state, kind, timestamps, etc.).
- `apply_state_summary`: `{ pending, applying, applied, failed, total_canary, total_promote }`.
- `gate_status`: `{ passed, last_eval_at, fail_reason }` (only meaningful in `soak`).
- Pointer to per-host detail at `/rollouts/:id/apply-state`.

### List Rollouts

```http
GET /api/v1/rollouts?scope_kind=product_variant&scope_ref=ship/linux
                    &state=in_flight|done|aborted
                    &since=<rfc3339>
                    &limit=50&cursor=<opaque>
```

Cursor-paginated. `state=in_flight` is a virtual filter for `state ∉ {done, aborted}`.

### Per-host ApplyState for a Rollout

```http
GET /api/v1/rollouts/:id/apply-state?state=failed&limit=100&cursor=...
```

Filterable by `state` so the UI can pull just the failures cheaply at fleet scale.

### Operator actions

```http
POST /api/v1/rollouts/:id/pause          → 200 OK or 409 (not pausable in current state)
POST /api/v1/rollouts/:id/resume         → 200 OK or 409
POST /api/v1/rollouts/:id/abort
  Body: { "reason": "<optional operator note>" }
                                         → 200 OK
POST /api/v1/rollouts/:id/fast-promote   → 200 OK or 409 (not eligible)
POST /api/v1/rollouts/:id/promote        → 200 OK or 409 (manual mode + gate_passed required)
```

All operator actions emit AuditEvents (§11).

## 10. Database schema

### `rollouts`

```sql
CREATE TABLE rollouts (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  scope_kind      TEXT NOT NULL CHECK (scope_kind IN ('product_variant','instance')),
  scope_ref       TEXT NOT NULL,
  config_id       INTEGER NOT NULL REFERENCES configs(id),
  prior_config_id INTEGER REFERENCES configs(id),
  rollout_kind    TEXT NOT NULL CHECK (rollout_kind IN ('phased','instant')),
  state           TEXT NOT NULL CHECK (state IN
                    ('validating','canary','soak','promoting','paused','done','aborted')),
  prev_state      TEXT,                 -- where Pause came from, for Resume
  canary_pct      INTEGER,              -- as requested (or default-resolved)
  canary_count    INTEGER,              -- as requested (or default-resolved)
  canary_size     INTEGER,              -- actual size at canary-start
  soak_seconds    INTEGER NOT NULL DEFAULT 300,
  gate_mode       TEXT NOT NULL CHECK (gate_mode IN ('auto','manual')) DEFAULT 'auto',
  gate_passed_at  TIMESTAMP,
  created_at      TIMESTAMP NOT NULL,
  created_by      TEXT NOT NULL,
  validated_at    TIMESTAMP,
  canary_at       TIMESTAMP,
  soak_at         TIMESTAMP,
  promoted_at     TIMESTAMP,
  done_at         TIMESTAMP,
  paused_at       TIMESTAMP,
  aborted_at      TIMESTAMP,
  abort_reason    TEXT
);

CREATE INDEX rollouts_scope_state ON rollouts (scope_kind, scope_ref, state);
CREATE INDEX rollouts_state_createdat ON rollouts (state, created_at DESC);

-- Partial index supporting live-config lookup at fleet scale
CREATE INDEX rollouts_scope_done ON rollouts (scope_kind, scope_ref, done_at DESC)
  WHERE state = 'done';
```

### `apply_state`

```sql
CREATE TABLE apply_state (
  rollout_id    INTEGER NOT NULL REFERENCES rollouts(id),
  instance_uid  TEXT NOT NULL,
  state         TEXT NOT NULL CHECK (state IN ('pending','applying','applied','failed')),
  is_canary     BOOLEAN NOT NULL,
  attempt_count INTEGER NOT NULL DEFAULT 0,
  applied_hash  TEXT,
  last_error    TEXT,
  pushed_at     TIMESTAMP,
  updated_at    TIMESTAMP NOT NULL,
  PRIMARY KEY (rollout_id, instance_uid)
);

CREATE INDEX apply_state_rollout_state ON apply_state (rollout_id, state);
CREATE INDEX apply_state_host ON apply_state (instance_uid, updated_at DESC);
```

### Live config resolution query

```sql
-- For a given (scope_kind, scope_ref):
SELECT config_id FROM rollouts
WHERE scope_kind = ? AND scope_ref = ? AND state = 'done'
ORDER BY done_at DESC
LIMIT 1;
```

The `rollouts_scope_done` partial index makes this O(log n) regardless of fleet size or rollout history depth.

### Recoverability

All state-machine transitions are DB writes. **Magpied has no in-memory state that affects Rollout correctness.** On restart, read all non-terminal rollouts and reconcile (see §12).

## 11. Audit events emitted

All events carry the common AuditEvent fields: `id, prev_hash, hash, type, actor, scope_ref, config_ref, host_ref, timestamp, payload`. Below: type-specific payload fields and emission triggers.

| Event type | Emitted when | Payload fields |
|---|---|---|
| `RolloutCreated` | Phased rollout is created (via Publish) | `rollout_id, rollout_kind=phased, canary_pct, canary_count, soak_seconds, gate_mode` |
| `RolloutInstant` | Instant rollout is created | `rollout_id` |
| `RolloutAdvanced` | Any state transition not covered by another event (incl. Resume) | `rollout_id, from_state, to_state` |
| `RolloutPaused` | Operator pause | `rollout_id, paused_from_state` |
| `RolloutAborted` | Aborted (operator-driven or auto) | `rollout_id, abort_reason ∈ {operator, canary_gate_failed, validate_failed, validate_timeout, no_canary_targets}, hosts_rolled_back` |
| `RolloutPromoted` | Manual-mode operator Promote | `rollout_id` |
| `RolloutFastPromoted` | Fast-Promote skipping Soak | `rollout_id, gate_status_at_skip ∈ {pending, passed, failed}` |

`actor` is bearer-token-validated identity for operator-triggered events, or `system` for auto-events (canary gate failure, validation timeout, all-applied transitions, etc.).

## 12. Failure handling

### Validate hangs

5s timeout. Rollout transitions to `aborted` with `abort_reason=validate_timeout`. AuditEvent: `RolloutAborted`.

### Canary push doesn't get OpAMP ack

If an OpAMP push to a canary host doesn't get an ack within 30s, the agent is treated as "push delivered, ack pending" — ApplyState stays in `applying`. If it stays in `applying` past 60s, gate evaluation counts it as not-applied (see §8). The row itself does not transition to `failed` — that's reserved for explicit collector-reported failures, so a delayed-ack host can still complete.

### Host disconnects during Canary

ApplyState stays `pending` (if no push went out yet) or `applying` (if push went out but no ack). At Soak window end, both states count as "not applied" for gate calculation. **Working assumption — see §13 #4.**

### Host fails apply (collector returns error from new config)

Agent reports `RemoteConfigStatus.Failed` with error message via OpAMP heartbeat. ApplyState transitions to `failed` with `last_error` populated. Gate evaluator counts this against the success rate.

### Magpied crashes mid-Rollout

On restart:

1. Read all rollouts in non-terminal states.
2. For each: read latest ApplyState rows for this rollout.
3. Re-derive expected state:
   - Was it in `validating`? If validate hadn't completed, abort with `abort_reason=validate_timeout` (the result is non-deterministic across a crash).
   - Was it in `canary` and all `apply_state` rows have left pending? Advance to `soak`.
   - Was it in `soak`? Resume gate evaluation. Soak clock resumes from where it was (using `soak_at` + ticks already accumulated; do not reset).
   - Was it in `promoting`? Re-issue pushes for any `pending` rows (idempotent — agent will see no-op if already on the right config).
   - Was it `paused`? Stay paused, await operator.
4. Resume normal phase advancement.

**Recovery test required (§14).**

### Hosts can't be reached at Rollout creation

If at canary-start, fewer hosts are connected than `canary_size = max(canary_count, ceil(canary_pct * connected))`, magpied uses what's there. The gate evaluates against actual canary size. If zero connected hosts in Scope at canary-start, abort with `abort_reason=no_canary_targets` (working assumption — see §13 #6).

### Operator deletes a host during a Rollout

The host's ApplyState row stays for audit. The host is removed from Scope membership. Gate evaluation continues against the original canary set including the deleted host's last-known state. If the deleted host was `pending` and is removed: it stays `pending` and counts as not-applied. (Operationally rare; calling out for completeness.)

## 14. Test surface

Tests are not v1.0 *features*, but the spec must be testable. v1.0 demonstrates correctness via:

### Unit tests (state machine)

- Every legal transition succeeds.
- Every illegal transition is rejected with the specific error.
- Pause from `validating` → 409.
- Fast-Promote from `canary` → 409.
- Resume from `paused` returns to correct `prev_state`.
- Promote (manual) without `gate_passed_at` → 409.

### Unit tests (gate evaluator)

- Apply success rate at 95% with 0 regressions → pass.
- Apply success rate at 94% → fail.
- Health regression with 100% apply rate → fail.
- Pending rows at window end count as not-applied.

### Unit tests (canary subset)

- Stable hash means same hosts canary repeatedly across rollouts.
- Subset size = `max(pct%, count)`.
- Subset size capped at connected fleet size if requested size exceeds it.

### Integration tests

- Full Phased rollout end-to-end: Create → Validate → Canary → Soak → Promote → Done. Assert audit event chain in correct order.
- Canary failure → auto-abort → rollback push reaches canary hosts → prior Config visible on those hosts.
- Validate failure → `aborted`, no host received the config (assert via OpAMP push log).
- Concurrent Publish to same Scope → second returns 409.
- Magpied restart mid-Rollout → state recovers from DB; Soak resumes correctly with remaining clock; auto-advance fires when conditions met.
- Instant Rollout → Promote without Canary/Soak; `RolloutInstant` event emitted.
- Manual-mode Soak → gate passes, stays in `soak`; operator Promote advances.

### Failure-mode tests

- Host disconnects during Canary push (before ack).
- Host disconnects during Soak.
- Host applies successfully, then goes unhealthy mid-Soak (regression detection).
- Operator pauses, resumes 30+ minutes later (gate re-evaluation).
- Operator aborts in `promoting` with 50% applied (rollback to those hosts).

### Performance tests

- Canary-start latency at 2000-host fleet target (canary size 100). Budget TBD in plan §8 #7.
- ApplyState aggregation query under load (used by every Rollout drawer poll, ~every 1–2s by N concurrent operators).
- Concurrent Rollouts across N different Scopes (no shared lock contention).
- Live-config resolution query at 100k rollout history depth (partial index keeps it constant-time).
