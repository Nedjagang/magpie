# Publish dialog — design spec (v1.0)

**Status:** Design only. **No code yet.** Open questions in §14 must be confirmed (or working assumptions accepted) before implementation begins.

**Scope of this document:** the UX from "operator clicks Publish in the Product drawer" through "rollout reaches Done (or aborts/pauses)." Covers visual structure, controls, defaults, validation states, live progress visualisation, and operator actions during an in-flight rollout. Does **not** cover the YAML editor itself (existing v0.2 affordance), the Rollouts list view (separate spec), or the Audit query surface.

**Reading order:** §1–§3 set up the entry path and structural pieces; §4–§7 go through the configuration controls; §8–§11 cover the publish-and-confirm flow + live progress; §12–§13 cover edge cases + accessibility; §14 calls out unanswered decisions; §15 is the test surface; §16 lists what's deliberately deferred.

**Supersedes / complements:** [`docs/v1.0-plan.md`](./v1.0-plan.md), [`docs/v1.0-rollout-spec.md`](./v1.0-rollout-spec.md). On any conflict the plan wins for scope/cut-line questions; the Rollout spec wins for state-machine semantics; this spec wins for publish-dialog UX.

---

## 1. Why this surface matters

Every state change in v1.0 flows through a Rollout (Rollout spec §1, plan invariant 1). The publish dialog is where the operator authors a Rollout — picking phasing, canary size, soak window, gate mode — and where blast radius is shown before the click that touches 2000 hosts. If this surface lies, gets out of operator's way too aggressively, or hides the cohort-write semantic, every other reliability guard in the system loses its teeth.

Two principles bind every decision in this spec:

1. **Defaults are safe.** An operator who hits Publish without touching anything else gets a Phased rollout with conservative canary and soak. To do anything riskier (Instant, skipping gates), the operator deliberately overrides — and that override is audit-logged.
2. **Blast radius is visible at the moment of commitment.** The operator never has to compute it, scroll for it, or hover-discover it.

---

## 2. Where the operator comes from

The publish dialog is reached from three entry points, all of which converge on the same dialog:

1. **Product drawer → "Publish" button** (most common). Operator edited YAML in the existing Edit tab, hits the button. The dialog opens populated with the Product's Scope and the YAML being published.
2. **Host drawer → "Open the config running here" → Product drawer → Publish** (host-first debugging). Same dialog as (1); the only difference is how the operator got there.
3. **Rollouts section → "Roll back to revision N"** (recovery). The dialog opens *pre-configured* with sensible-for-rollback defaults: same Scope, the prior Config's YAML, **Instant** pre-selected (with the amber warning showing) since the operator is recovering from a failed change. The operator can flip to Phased if they prefer to canary the rollback.

All three paths converge on the same dialog component. There is no "publish for normal" vs "publish for rollback" duplication.

## 3. Structural anatomy

The publish dialog is a **two-pane modal** that opens as an overlay in the Product drawer (does not navigate away from the editor — operator can hit Cancel and return to YAML state intact).

Schematically:

```
┌──────────────────────────────────────────────────────────────┐
│ Publish to ship/linux                            [✕ Cancel]  │
│ ──────────────────────────────────────────────────────────── │
│                                                              │
│ ┌──── BLAST RADIUS ───────────┐  ┌──── ROLLOUT ────────────┐ │
│ │                             │  │                         │ │
│ │  ship/linux                 │  │  ◉ Phased (recommended) │ │
│ │  12 hosts targeted          │  │  ○ Instant              │ │
│ │  12 healthy · 0 drifting    │  │                         │ │
│ │  Last published 4h ago      │  │  ┌─ Canary ───────┐     │ │
│ │                             │  │  │ 5 % or N=10    │     │ │
│ │  Validation                 │  │  └────────────────┘     │ │
│ │  ✓ structural               │  │                         │ │
│ │  ✓ semantic (otelcol)       │  │  Soak: 5 min            │ │
│ │  ✓ component allowlist      │  │  Gate: auto             │ │
│ │                             │  │                         │ │
│ └─────────────────────────────┘  └─────────────────────────┘ │
│                                                              │
│  Tags (optional): post-incident-fix                          │
│                                                              │
│ ──────────────────────────────────────────────────────────── │
│  This will start a rollout to 12 hosts.                      │
│  Canary 1 host (8%) → soak 5 min → promote remaining 11.     │
│                                                              │
│                              [ Cancel ]  [ Start rollout ▸ ] │
└──────────────────────────────────────────────────────────────┘
```

Three structural zones, each load-bearing:

- **Header.** Names the destination Scope explicitly. No ambiguity about what's being published where. This is the first thing the operator's eye lands on; a misclick into the wrong cohort drawer becomes obvious before any clicks below.
- **Left pane — Blast radius + validation.** Read-only. Updates live (host count is current, drifting count is current, validation status updates as YAML changes upstream).
- **Right pane — Rollout configuration.** All operator-tunable controls.
- **Footer — Sentence summary + action.** Plain-English description of what's about to happen + the single primary action.

The pre-publish summary in the footer is a re-statement of the same data shown above, in narrative form. Rationale: the operator scans visual structure for state but reads the footer for verification before clicking. Two redundant channels for the load-bearing fact.

## 4. Rollout kind (Phased vs Instant)

A radio with two options, defaulting to **Phased**.

### Phased (default)

The recommended path. Selected by default. No friction.

### Instant

Selecting Instant **collapses** the canary/soak/gate controls (they're not meaningful for an Instant rollout) and replaces them with an amber warning banner:

```
⚠ Instant rollout
This bypasses Canary and Soak. All 12 hosts will receive the config in
~2 seconds. The audit log will distinguish this from a Phased rollout
(event type: RolloutInstant). Use only for emergencies.
```

The "Start rollout" button is still primary (we don't gate Instant behind a secondary confirmation — the warning text is the friction). Instant is a deliberate operator choice; if they read the banner and click anyway, that's the audit trail.

Instant remains visible in the dialog even when not selected — it's a peer option, not a hidden expert mode. Operators looking for it shouldn't need to hunt.

## 5. Canary controls

Visible only when Phased is selected. Two interlocking inputs:

```
┌─ Canary ─────────────────────┐
│  ┌──────┐    ┌────────────┐  │
│  │  5 % │ or │ at least 10│  │
│  └──────┘    └────────────┘  │
│  Effective today: 1 host     │
└──────────────────────────────┘
```

- **Percent.** 1–50, default 5. Below 1% would mean zero canary on a fleet < 100; we don't allow that. Above 50% would mean canary > rest, which defeats the purpose; we cap.
- **Count.** Numeric, default 10. The actual canary size is `max(ceil(pct% × connected_hosts), count)`. The "Effective today" line below shows the resolved size against the *current* connected fleet — operators can see exactly how many machines they'd hit.
- **"Effective today"** updates live as the operator changes either input or as connected hosts change (poll every 2.5s same as elsewhere).

For Scopes with very small connected fleets (e.g. 3 hosts), the count `10` would saturate the whole fleet. The dialog handles this explicitly:

```
Effective today: 3 hosts (entire connected Scope — no Promote phase)
```

The rollout still goes through Validate → Canary → Soak → (no Promote, no rest to promote to) → Done. Soak still applies. This is correct behavior — small fleets just have Canary == Promote in practice.

If the connected fleet is **zero**, the input is disabled and the Start button is greyed with the tooltip "No connected hosts in ship/linux." The Rollout spec's `no_canary_targets` abort is the engine-side mirror; the dialog prevents the Rollout from being created in the first place.

## 6. Soak window

Single numeric input with unit selector:

```
Soak: ┌───┐ minutes ▾
      │ 5 │
      └───┘
```

- **Default 5 minutes** (300s — matches Rollout spec §4.3).
- Range: 1 minute — 60 minutes for v1.0. Outside that range either reject (< 1) or warn (> 60: "long soaks tie up the rollout slot for this Scope; consider 30 min max").
- Unit selector: `seconds | minutes`. Default minutes. Operators authoring a 30-second soak get an explicit display warning ("very short — gate may not have time to detect health regression").

The soak window doesn't apply to Instant rollouts; the control is hidden when Instant is selected.

---

## 7. Gate mode

Two-state toggle: **Auto** (default) / **Manual**.

```
Gate: ◉ Auto      ○ Manual
       advances when canary is healthy.
```

Auto means: gate passes → automatic transition to Promote. The operator does not need to be present.

Manual is the change-management option: when the canary gate passes, the rollout sits in `soak` with `gate_passed_at` set, awaiting an explicit operator Promote click. The dialog shows:

```
Gate: ○ Auto      ◉ Manual
       holds at "ready to promote" until you Promote.
```

Manual mode is appropriate when (a) compliance requires a second operator to sign off on production rollouts, or (b) the operator wants to inspect canary applies before promoting. Auto is the v1.0 default — confirmed in Rollout spec §13 #12.

Manual mode does **not** override auto-rollback on canary failure (Rollout spec §6). Canary failure always aborts; manual mode only gates the *promote* step.

## 8. Live blast-radius display

The left pane is informational but load-bearing. Every line is a fact the operator needs *before* committing.

### Host count + health

```
12 hosts targeted
12 healthy · 0 drifting
```

- **Targeted.** Currently connected hosts that match this Scope. Polled live.
- **Healthy.** Subset that's reporting healthy via OpAMP.
- **Drifting.** Subset whose `applied_config_hash` differs from the Scope's current live Config (per Rollout spec §4.3 — drift detection).

Drift > 0 surfaces a yellow note: `⚠ 3 hosts drifting from current config` — operators investigating a rollout might want to know they're not starting from a clean state. (Drifting hosts will receive the new config like everyone else; the note is informational.)

### Last published

```
Last published 4h ago by alice@aptean.com
```

Fetched from the most recent Done rollout for this Scope. If there's never been a successful publish (brand-new Scope), the line reads:

```
First publish to this Scope.
Hosts on ship/linux currently fall back to default/default.
```

That phrasing is intentional: the first publish to a new Scope changes the host's config from default fallback to the new YAML. Operators forget this; surfacing it prevents the "wait, why did the new hosts pick this up immediately" surprise.

### Validation status

```
Validation
✓ structural
✓ semantic (otelcol)
✓ component allowlist
```

Each checkmark corresponds to a check that ran on the YAML in the editor (debounced; runs as the operator edits). If any fails, the dialog refuses to start the rollout — the Start button is disabled with a tooltip pointing at the failing check.

A failing semantic validation, for instance:

```
Validation
✓ structural
✗ semantic (otelcol): processor "memory_limiter" not configured
✓ component allowlist
```

→ Start button disabled. Operator returns to editor (Cancel button takes them back without losing the YAML or the rollout config they've set up).

## 9. Confirmation flow

There is **no second confirmation dialog after Start rollout**. The publish dialog itself is the confirmation — it shows blast radius, rollout shape, and the explicit primary button. A separate "Are you sure?" modal would be theatre after this much information is already on screen.

The single exception is **Instant rollout when targeting > 100 hosts**. At that scale, "Instant" really means "100+ hosts in 2s with no canary," and we add a one-line check:

```
About to instantly publish to 1,840 hosts. No canary, no soak.
                                       [ Cancel ]   [ Confirm ▸ ]
```

This is a thin checkbox, not a full modal. Threshold (100 hosts) is conservative enough that small-fleet operators never see it; large-fleet operators get the brake. Open question §14 #3 — confirm or revise.

## 10. Live progress UI

Once the operator clicks "Start rollout," the dialog **transforms in place** (does not close, does not navigate away — Rollout spec invariant: "no navigation away after publish"). The configuration controls fade out; in their place a live phase-progress view fades in:

```
┌──────────────────────────────────────────────────────────────┐
│ Rolling out to ship/linux                            #r1247  │
│ Started 12s ago by alice@aptean.com  ·  Phased  ·  Auto      │
│ ──────────────────────────────────────────────────────────── │
│                                                              │
│   Validate ✓ ─── Canary ▸ ─── Soak ─── Promote ─── Done      │
│                  ●                                           │
│                                                              │
│   Canary: 1/1 applied · 0 failed                             │
│   Health: 1/1 healthy                                        │
│                                                              │
│   ┌─ Canary hosts ──────────────────────────────┐            │
│   │ ● db-prod-7      applied   3s ago           │            │
│   └─────────────────────────────────────────────┘            │
│                                                              │
│ ──────────────────────────────────────────────────────────── │
│ [ Pause ]  [ Abort ]                  [ Fast-promote ▸ ]     │
└──────────────────────────────────────────────────────────────┘
```

### Phase indicator

Five phase pills, each showing one of three states:

- **Done** — checkmark, green.
- **Active** — animated dot, accent color.
- **Pending** — outline only, muted.

The phase indicator never goes backward except on Abort (where the rollout transitions to `aborted` and the pills collapse to a single red `Aborted` pill with the abort_reason).

### Per-phase details

Below the phase pills, a section shows what's happening *in the current phase*:

- **Validate.** Spinner + "Running structural + semantic checks…". Usually < 2s, almost never visible to operators because Validate runs before Canary's first push.
- **Canary.** Two metric lines: applied/total, healthy/total. A collapsible list of canary hosts with their per-host ApplyState (pending / applying / applied / failed) and the timestamp of the last update.
- **Soak.** Countdown timer ("4m 12s remaining"), gate evaluation status ("Gate: ✓ apply rate 100% · ✓ no health regression"). Updates every 10s (same cadence as gate evaluation in Rollout spec §4.3).
- **Promote.** Same shape as Canary metrics but for the rest of the fleet. Per-host list paginated/virtualized at 2000-host scale (cf. Fleet view spec).
- **Done.** Summary card: total applied, total failed (if any), total time, "Hosts now running [config-name v9]." A `[ View full rollout ]` link goes to the Rollouts section drawer for this rollout.

### The host list at scale

In the Promote phase at 2000-host scale, listing every host is impractical even virtualized. Default view shows aggregate counts only with three drill-in tabs:

```
   Promote: 1,840/2,000 applied · 12 failed · 148 pending

   ┌─ Failed (12) ─ Pending (148) ─ Applied (1840) ─┐
   │ host-prod-93     processor "logs/foo" missing  │
   │ host-prod-127    timeout — agent unreachable   │
   │ host-prod-241    processor "logs/foo" missing  │
   │ ...                                            │
   └────────────────────────────────────────────────┘
```

Failed gets pride of place (top tab, default selection when there's any failed) — operators investigating a rollout almost always want to see what's failing first. Pending is the next-most-interesting (hosts that should have applied by now but haven't reported back). Applied is reassuring background.

## 11. Operator controls during rollout

Three buttons in the footer of the live-progress view, plus a Manual-mode-only fourth:

### Pause

- Visible during `canary`, `soak`, `promoting`. Disabled during `validating` (Validate is fast and atomic — can't be paused mid-flight) and `done`/`aborted`.
- Click → calls `POST /api/v1/rollouts/:id/pause`. Phase indicator's active dot stops animating; a pause overlay covers the metrics area:

```
   Canary ▸ ─── Soak ⏸ ─── Promote ─── Done
                  ●
                  
   Paused 1m 3s ago by alice@aptean.com
   Soak clock: 4m 28s remaining (preserved)
   
   [ Resume ]  [ Abort ]                  [ Fast-promote ▸ ]
```

The soak clock state is preserved across Pause (per Rollout spec §3 — "pause stops the clock"). Resume picks up where it left off and re-evaluates the gate immediately (also per Rollout spec).

### Abort

- Visible in any non-terminal state.
- Click → confirm dialog with explicit blast radius:

```
   Abort rollout to ship/linux?
   
   Hosts on the new config will be rolled back to the prior config:
     [config-name v8] (3a91…)
   
   ▸ 1 canary host will receive the rollback push.
   ▸ 0 promote hosts received the new config.
   ▸ Rollout will be marked aborted (RolloutAborted event).
   
                                    [ Cancel ]   [ Abort rollout ]
```

The "Abort rollout" button is danger-styled. Confirmation is required because Abort is a state-changing action that touches hosts (rollback push). On confirm: `POST /api/v1/rollouts/:id/abort` with the operator's reason (text input, optional).

### Fast-Promote

- Visible during `soak` and `paused` (when `prev_state=soak`). Disabled during `canary` and `validating` (gates haven't been evaluated yet — nothing to skip).
- Click → confirm dialog:

```
   Fast-promote to all 11 remaining hosts?
   
   This skips the remaining soak window. Canary applies are currently
   passing (1/1 applied, 0 health regressions detected), so this is
   reasonable, but the safety window will not run to completion.
   
   Audit log will record RolloutFastPromoted with the gate state at
   time of skip.
   
                       [ Cancel ]   [ Fast-promote ▸ ]
```

The confirm is informational, not blocking. The current gate state is shown (passing / failing / still evaluating) so the operator knows what they're skipping. If gate is currently *failing*, a yellow warning is added: "Canary is currently failing the gate. Fast-promoting will deliver a known-failing config to the rest of the fleet. Reconsider."

### Promote (Manual mode only)

- Visible during `soak` only when (a) `gate_mode=manual` AND (b) `gate_passed_at` is set (the gate passed but auto-promote is held).
- Click → simple confirm: "Promote to remaining 11 hosts?" → `POST /api/v1/rollouts/:id/promote`.

## 12. Validation and error states

### Validate failed

If semantic or structural validation fails after the operator clicked Start rollout (rare — usually validation runs in the editor as the YAML is typed, but possible if the editor was bypassed via API or the validation rules changed mid-edit):

```
   ✗ Rollout aborted at Validate
   semantic: processor "memory_limiter" not configured
   
   No host received this config.
   
   [ Edit YAML ▸ ]    [ Close ]
```

The "Edit YAML" button takes the operator back to the Product drawer's editor with the YAML still loaded. Cancel is "Close" because there's nothing to cancel (the rollout already terminated).

### Canary gate failed → auto-abort

```
   ✗ Rollout aborted by canary gate
   Apply success rate: 0/1 (1 failed: db-prod-7)
   Failure: processor "logs/foo" not configured
   
   The 1 canary host has been rolled back to the prior config.
   
   [ Edit YAML ▸ ]    [ Close ]
```

The dialog stays open in this terminal state until the operator dismisses it — they should see *why* the rollout aborted, not have it disappear and force them to chase the audit log.

### Pause for a long time

If the rollout has been paused for > 30 minutes, the dialog adds a gentle prompt:

```
   ⓘ This rollout has been paused for 33 minutes.
     Paused rollouts hold the in-flight slot for this Scope; new
     publishes to ship/linux will return 409 until you resume or abort.
```

This isn't blocking, just a reminder — long paused rollouts are operationally costly because the Scope is locked.

### Network failures during operator action

If the operator clicks Pause/Abort/Fast-Promote and the API call fails (network blip, magpied unreachable), surface inline:

```
   ⚠ Could not reach magpied: <reason>
   The rollout is unaffected. Try again, or check magpied status.
   
   [ Retry ]
```

The rollout itself continues — it's running on the server side. Only the operator's *control intent* failed to land. Retry sends the same call again; rollout state is idempotent for these operations (Pause-when-paused = no-op, etc.).

## 13. Cross-product guard interaction

The publish dialog as described targets a single Scope. The cross-product guard (Rollout spec §3 concurrency rule, plan §5 reliability guard) prevents the operator from creating a rollout for a Scope that already has an in-flight Rollout.

Two surfaces in the publish dialog:

### 1. Pre-publish detection

When the dialog opens, it queries `GET /api/v1/rollouts?scope=ship/linux&state=in_flight`. If a rollout is already in flight, the dialog opens in a banner-warning state:

```
   ⚠ A rollout to ship/linux is already in flight (#r1246, soak phase).
   Publishing now requires aborting that rollout first.
   
   [ View in-flight rollout ▸ ]    [ Cancel ]
```

The "Start rollout" button is replaced by "View in-flight rollout" (which navigates to the Rollouts section's drawer for that rollout). Operator can Abort there, then return here to publish.

This is preferable to a 409 from `POST /rollouts` after the operator already configured everything — that wastes their work.

### 2. Race condition

Even with pre-publish detection, two operators publishing simultaneously can race past it. The 409 from the API is still possible. In that case the dialog handles the 409 inline:

```
   ⚠ Another rollout to ship/linux started while you were configuring.
   Rollout #r1248, instant phase, started 4s ago by bob@aptean.com.
   
   [ View ▸ ]    [ Cancel ]
```

Operator can wait for bob's rollout to finish, then publish; or abort bob's if they have authority and reason.

## 15. Test surface

Tests are not v1.0 *features*, but the spec must be testable.

### Component tests (UI-level)

- Renders with default values (Phased, 5%/N=10, 5 min soak, auto gate).
- Switching to Instant collapses canary/soak/gate controls + shows amber banner.
- Switching back to Phased restores defaults (or operator's last values — open question §14 not listed).
- Validation failure disables Start button with correct tooltip.
- "Effective today" canary size updates as percent/count change and as connected fleet count changes.
- Cross-product guard banner appears when in-flight rollout exists for the Scope.
- Pre-publish 409 from API surfaces the inline race-condition message.

### Integration tests (with magpied)

- Publish flow: Edit YAML → open dialog → Start → assert `POST /api/v1/rollouts` payload matches dialog state.
- Live progress: poll cadence drives ApplyState aggregate updates.
- Pause / Abort / Fast-Promote / Promote each map to the right API endpoint and update dialog state on response.
- Operator browser close + reopen reattaches to in-flight rollout.

### Failure-mode tests

- Validation failure post-Start (rare race but possible) → terminal-aborted state shown clearly.
- Canary gate failed → auto-abort → terminal state shown with failure detail.
- Network error on operator action → inline retry, rollout state unaffected.
- Pause > 30min → inline prompt.

### Accessibility tests

- Dialog is keyboard-navigable (tab order: kind → canary % → canary count → soak → gate → tags → Cancel → Start).
- Phase indicator state changes announce via aria-live="polite" (not assertive — operators reading along don't need every 10s gate-eval poll narrated).
- Color isn't the only signal: phase pills have icons + text; failed/applied/pending tabs have counts in the label.

## 16. Out of scope for this spec

These are real concerns, deliberately not addressed here:

- **The Rollouts list view** (separate spec). The publish dialog navigates to it via "View in-flight rollout" links but doesn't render it.
- **The Audit query surface** (separate spec). The publish dialog produces audit events but doesn't show the audit log.
- **The Fleet view's per-host drilldown.** The publish dialog's Failed/Pending tabs show top-N hosts; the full fleet-scale list is the Fleet view spec's territory.
- **Per-host canary selection** ("pick these specific hosts as canary"). v1.x; not in this spec.
- **Multi-Scope publish** ("publish to ship/linux + ship/windows in one rollout"). v1.x; for v1.0 the operator opens two dialogs and starts two rollouts.
- **Webhook / Slack notifications on rollout state changes.** v1.x.
- **Rollout templates / saved rollout configurations.** v1.x.
