# Host drawer — design spec (v1.0)

**Status:** Design only. **No code yet.** Open questions in §15 must be confirmed (or working assumptions accepted) before implementation begins.

**Scope of this document:** the slide-in drawer that opens when an operator clicks a row in the Fleet view. Covers diagnostic header, Scope resolution panel, in-flight rollouts strip, health + apply state, recent activity (host-scoped), label override (re-assign cohort), per-host config override (Shape 1), version drift, attributes dump, and the destructive delete affordance. Does **not** cover the Fleet view itself (separate spec), publish dialog (separate spec), or the audit query surface (separate spec — though this drawer embeds a small recent-activity slice that derives from the same data).

**Reading order:** §1–§3 frame why this surface matters and how it's structured; §4–§7 walk through the upper diagnostic-led panes; §8–§11 cover the operator-action panes; §12 covers state + edge cases; §13–§17 cover deferrals, accessibility, open questions, test surface.

**Supersedes / complements:** [`docs/v1.0-plan.md`](./v1.0-plan.md), [`docs/v1.0-rollout-spec.md`](./v1.0-rollout-spec.md), [`docs/v1.0-publish-dialog-spec.md`](./v1.0-publish-dialog-spec.md), [`docs/v1.0-fleet-view-spec.md`](./v1.0-fleet-view-spec.md). On any conflict the plan wins for scope/cut-line questions; the relevant spec wins for its surface; this spec wins for host-drawer UX.

---

## 1. Why this surface matters

The host drawer is **the surface where reads-end-meets-writes-begin**. Plan §4 made the IA host-first because the operator's debugging mental model is "this machine is broken — what's wrong with it." The drawer is what answers that question. From it, the operator decides whether to fix the cohort (open the Product drawer + publish a fix), override the host (Shape 1: instance-scoped publish), re-assign the host to a different cohort (label override), or delete a stale row.

Three principles bind every decision in this spec:

1. **Diagnostic before meta.** The first thing the operator's eye lands on is *what's wrong* (or "everything's fine"), not section headers, not ID fields. v0.2 led with the host name in big serif type and put "config status: failed" three sections down. v1.0 inverts that — diagnostic at the top, meta below.
2. **Scope resolution is explicit.** A v1.0 host's effective config can come from one of three places (instance override, product+variant scope, default fallback). The drawer states which, names the Scope ref, and shows the resolution chain. No operator should have to compute this from labels + override flags + cohort metadata.
3. **Every action names its blast radius.** Re-assigning a cohort moves this *one* host. Overriding the host's config writes a Shape 1 override that affects this *one* host. Editing the cohort config affects *N* hosts (cohort size). Each affordance shows N before the click, matching the Reliability principle from publish dialog §1 §9.

---

## 2. Where the operator comes from

Three entry paths, all converging on the same drawer:

1. **Fleet view row click** (most common). Operator scans Fleet, finds a failed/drifting row, clicks. Drawer slides in from the right. Filter state in the URL is preserved so closing the drawer returns to the same Fleet view.
2. **Deep link.** A URL like `/fleet/h/abc-123-def` opens Fleet with the drawer already open on host `abc-123-def`. Useful for sharing "this specific machine is broken" across operators (Slack-friendly).
3. **Audit event click.** A `HostDeleted` / `LabelChanged` / `OverrideApplied` event in the Audit section has its `host_ref` clickable; click opens the host drawer for context. (If the host has been deleted, the drawer opens in a "not found" state — see §12.)

The drawer is dismissed by Cancel button, Escape key, scrim click, or back button (deep link unwinds the host portion of the URL while keeping the Fleet filter state).

## 3. Structural anatomy

The drawer is a **right-side slide-in** at width ~720px on desktop, 100vw on mobile. Vertical scroll within the drawer; the Fleet view dim-scrim behind it.

Eight zones, top to bottom:

1. **Header** (host name + close).
2. **Diagnostic block** (✗/●/✓ + reason + primary action buttons).
3. **Scope resolution** (the v1.0 keystone — explicit chain + applied/live hash).
4. **In-flight rollouts** (conditional — hidden when none).
5. **Health summary**.
6. **Recent activity** (host-scoped audit slice).
7. **Operator actions** (Re-assign cohort, Override config, version drift).
8. **Attributes dump + danger zone**.

The diagnostic block + Scope resolution + in-flight rollouts together form the "what's happening with this host" section. Everything below is "what can I do about it" plus reference data. A reader who only needs to triage the host should not need to scroll past the Scope resolution panel.

## 4. Diagnostic header

The first thing the operator's eye lands on after the host name. Three states:

### Healthy + applied (the boring case)

```
✓ Applied
Last applied 3s ago.

[ Open the config running here ▸ ]   [ Re-assign cohort ]
```

Even in the boring case, the primary action remains visible — the operator might be there to read the config, not fix anything.

### Failed apply

```
✗ Failed apply
processor "memory_limiter" not configured

[ Open the config running here ▸ ]   [ Re-assign cohort ]
```

The error from `config_error` is shown verbatim. If multi-line, it's rendered in a `<pre>` block at typical reading width.

### Drifting

```
⚠ Drifting
Applied 7c02ab94, scope's live is 3a91…  Drift since 12s ago.

[ Open the config running here ▸ ]   [ Re-assign cohort ]
```

Drift = `applied_config_hash ≠ live_config_hash for the host's resolved Scope, AND no in-flight ApplyState explains the gap`. The drift since timestamp is when the divergence was first detected.

### Other states

- **Unhealthy** (OpAMP `health=false` regardless of config): `✗ Unhealthy — agent reports it's not running collector pipelines correctly`.
- **Pending** (host registered, never reported applied state): `⏳ Pending — agent connected but hasn't applied a config yet`.
- **Applying** (in-flight ApplyState row exists): `▸ Applying ship-linux-v9 (3s ago)`.

The diagnostic line determines the dot color (✓ green, ⚠ yellow, ✗ red, ⏳ neutral) and is repeated as the row's status pill in the Fleet view — so what an operator sees in the table matches what they see in the drawer.

## 5. Scope resolution panel

This is the v1.0 keystone of this surface. It states **explicitly** which of the three Scope shapes is producing the host's effective config.

### Layout — three-stop chain

```
Scope resolution
┌─ instance: (none) ── product+variant: ship/linux ── default: default/default ─┐
│                            ↑ in force                                          │
└──────────────────────────────────────────────────────────────────────────────┘
Live config: ship-linux-v9 (7c02ab94)
Applied:     3a91…  ↗  drift since 12s ago
```

The chain shows the resolution priority left-to-right: instance > product+variant > default. The "↑ in force" arrow points at the Scope that's actually producing the live config for this host. Three cases:

1. **In force = instance.** This host has a per-host (Shape 1) override. The chain shows `instance: db-prod-special-override-v3 ── product+variant: ship/linux ── default: default/default`, with the arrow pointing at instance. The product+variant entry is greyed (it's not in force, but the operator should still see what would take over if the override were cleared).

2. **In force = product+variant.** Most common. Chain shows the host's effective `(product, variant)` cohort. The arrow points there. The instance entry shows `(none)` greyed.

3. **In force = default.** Brand-new host, or a host whose product+variant has no published Config and there's no instance override. Chain shows the arrow at `default/default`. Both other entries are greyed/empty.

### Live + applied lines

Below the chain, two information lines:

- **Live config.** The Config row currently in force for this Scope (i.e. the config the host *should* be running). Format: `<config-name> (<short-hash>)`. Click → opens the Product drawer at this Scope, scrolled to the History tab with this revision highlighted.
- **Applied.** What the host's last OpAMP heartbeat reported as `applied_config_hash`. Matching = no decoration. Drifting = `↗` icon + "drift since N ago". Failed = `✗` icon + last error (already shown in the diagnostic block, repeated here for the operator who scrolled past it).

### Why this layout

The alternative — a single "Effective scope: ship/linux" line — buries the resolution chain. At v1.0 with Shape 1 overrides in play, an operator looking at a host with a per-host override needs to see *both* the override and what would replace it on clear. The chain is direct.

The arrow + greyed inactive entries also sets up the **"clear override"** affordance from §10.2 — the operator who clears the override sees their action move the arrow from instance to product+variant.

## 6. In-flight rollouts (conditional)

Hidden when the host is not part of any in-flight Rollout. Visible (typically one card; rarely more) when it is.

```
In-flight rollout (1)
┌─────────────────────────────────────────────────────────────┐
│ #r1247 · ship/linux · Soak (4m 12s)                         │
│ This host is in the canary subset · failed apply 12s ago    │
│                                          [ View rollout ▸ ] │
└─────────────────────────────────────────────────────────────┘
```

Each card shows:

- **Rollout id + Scope + phase** (with a phase pill).
- **This host's role**: `canary subset` (selected as one of the canary hosts) or `promote subset` (post-soak fan-out).
- **This host's apply state**: `pending`, `applying (3s ago)`, `applied (3s ago)`, or `failed apply 12s ago` (with the brief reason).
- **View rollout button** — opens the Rollouts section drawer for that rollout.

Multiple in-flight rollouts can target the same host only in narrow edge cases (e.g. an instance Rollout and a product+variant Rollout both touching this host because one supersedes the other on next resolve). Spec is conservative: render up to 3 cards stacked, "+N more" if higher.

This section's data comes from `GET /api/v1/rollouts?state=in_flight` filtered to those that have an `apply_state` row for this `instance_uid`. The Fleet view's poll cadence (2.5s) drives updates here too.

## 7. Health summary

Compact, four lines, no buttons. Reads at a glance.

```
Health
 ● healthy: yes
 ● OpAMP status: running
 ● connected 2h ago
 ● last seen 2s ago
```

- **healthy**: `yes` / `no` / `not yet reported` (three-state, per Rollout spec §3.1).
- **OpAMP status**: the `last_status` string the agent sent (`running`, `stopped`, `loading`, etc.).
- **connected**: time since `connected_at` (relative; absolute on hover).
- **last seen**: time since `last_seen`. If > 5 min ago, this line goes amber; if > 1 hour, it goes red.

Last seen is the freshness signal — if it's stale, the rest of the data in the drawer may be stale too. The amber/red color is a passive cue, not a banner. (Compare to v0.2 which had no freshness indicator at all.)

## 8. Recent activity (host-scoped)

A small embedded slice of the audit log filtered by `host_ref = <this instance_uid>`. Shows the last 5 events; full history available via "View all" → Audit section pre-filtered to this host.

```
Recent activity (this host)
 • RolloutCreated  ship/linux v9    12s ago  alice@aptean
 • LabelChanged    ship/linux       4d ago   bob@aptean
 • OverrideCleared                   8d ago   alice@aptean
 • [ View all events for this host in Audit ▸ ]
```

Event types this surface highlights:

- `RolloutCreated`, `RolloutAdvanced`, `RolloutPaused`, `RolloutAborted`, `RolloutPromoted`, `RolloutFastPromoted`, `RolloutInstant` — only when this host was in the Rollout's canary or promote target set.
- `OverrideApplied`, `OverrideCleared` — instance-scoped overrides.
- `LabelChanged` — relabel events.
- `HostDeleted` — won't show here since the drawer wouldn't open on a deleted host, but appears in the URL-scoped audit query.

If the audit slice has zero events for this host, the section shows: "No recorded activity. Events from before v1.0 may not be host-tagged." (Pre-v1.0 audit rows don't have `host_ref` populated; the §10 in audit spec covers what's queryable retroactively.)

## 9. Re-assign cohort (label override)

This is the v0.2 carryover affordance — operator overrides the host's self-reported `(product, variant)` so the host gets a different Scope's config without SSHing to change env vars.

### Initial state — no override active

```
Re-assign to a different cohort               [ Re-assign ]
```

Click → expands inline:

```
Re-assign to a different cohort
   ┌──────────────────┐  ┌──────────────────┐
   │ Product:    ship │  │ Variant:   linux │
   └──────────────────┘  └──────────────────┘
   This host advertises ship/linux. Override saves a label-only
   reassignment — config follows the new cohort on next resolve.

   [ Cancel ]                              [ Apply override ]
```

The two inputs are pre-populated with the host's *advertised* labels (so the operator clearly sees what they're changing from). Datalist autocomplete on Product (drawn from existing Products in the fleet).

Click "Apply override" → `PUT /api/v1/agents/{uid}/labels` with the new product+variant. On success, the chain in §5 updates: the product+variant entry shows the new value, the host's effective Scope flips, and the next resolve push delivers the new Scope's config. An `OverrideApplied` audit event is emitted.

### Override active state

When a label override exists, the section shows:

```
Override active                          [ Update ]  [ Clear override ]
   advertised: ship/linux  →  effective: pay/linux  (4d ago, by bob@aptean)
```

The "advertised → effective" line is the diff. "Clear override" → confirm dialog → `DELETE /api/v1/agents/{uid}/labels` → emits `OverrideCleared`.

## 10. Override this host's config (Shape 1)

This is **new in v1.0**. Allows the operator to give this one host a custom config that's distinct from any product+variant Scope's config — instead of moving the host to a different cohort (label override), the host gets its own per-host Config.

### 10.1 No override active

```
Override this host's config              [ Override ▸ ]
   This host follows ship/linux's config. To give it a custom config
   that's distinct from the cohort, override here.
```

Click → opens the **publish dialog** (already spec'd in `v1.0-publish-dialog-spec.md`) with `scope_kind=instance, scope_ref=<this instance_uid>` pre-set. The dialog's blast radius pane reads:

```
   ship/linux → instance:db-prod-7
   1 host targeted
   1 healthy · 0 drifting
```

…because instance scope targets exactly one host. Canary defaults still apply (canary size = 1 = whole "fleet" of this Scope), but Soak still runs (catches a failing apply before considering the rollout Done).

### 10.2 Override active

When this host has an active per-host config override, the section shows:

```
Override active                              [ View ]  [ Clear override ]
   instance:db-prod-7 v3 (a8f201ce)
   Drift from product+variant: 3 days  ·  ⚠ ship/linux is on v9
```

The "Drift from product+variant" line is the load-bearing nudge: operators forget overrides exist; surfacing how stale they are vs the cohort they bypass keeps that context visible.

**Clear override** → confirm dialog with **YAML diff**:

```
   Clear override for db-prod-7?
   
   This will move db-prod-7 from instance scope back to product+variant
   (ship/linux). The host will receive ship/linux v9 on next resolve.
   
   Diff (override → ship/linux v9):
   - receivers: [otlp]
   + receivers: [otlp, hostmetrics]
   ...
   
                                  [ Cancel ]   [ Clear override ]
```

The diff is the substantive reliability beat — operators see what's about to change before clearing. (This matches plan §5: "Override clear shows YAML diff before confirming.")

Clear emits an `OverrideCleared` audit event with the diff content in `payload_json` for forensic recall.

**View** → opens the Product drawer at `instance:db-prod-7`, scrolled to the override config, with full History tab.

### 10.3 Why this and label override are separate

A label override moves the host to a different existing cohort's config — works when "what this host needs" is already authored elsewhere. A config override creates a new per-host config — works when the host genuinely needs something nobody else does. Conflating them is what trapped v0.2 operators into the "spin up a private cohort with one host" workaround. v1.0 makes both explicit and gives each its own affordance.

The two can coexist on a single host (label override moves it to pay/linux; config override gives it a custom config that resolves before pay/linux's). The Scope resolution panel §5 makes the chain visible regardless.

## 11. Agent version drift (v0.2 carryover)

v0.2 already had this surface. Carries forward unchanged in shape:

```
Agent version drift
┌─────────────────────────────────────────────────────────────┐
│ ⚠ Manual upgrade available                                  │
│ Host reports v0.2.0  ·  control plane is v1.0.0             │
│                          [ Download windows-amd64.zip ▸ ]   │
└─────────────────────────────────────────────────────────────┘
```

Visible only when `host.attributes["service.version"]` differs from the magpied version. Hidden when versions match or when the host hasn't reported a version yet.

The download button uses the existing `/api/v1/releases/<os>/<arch>` endpoint. OS/arch come from the host's reported attributes. If the host hasn't reported them, button is replaced with "OS/arch not reported; see docs/onboarding.md."

## 12. State + edge cases

### Pending host (registered, no heartbeat yet)

```
⏳ Pending — agent connected but hasn't applied a config yet
```

The Scope resolution panel still shows the chain (resolution happens server-side from advertised labels even before first apply). In-flight rollouts strip is hidden (no ApplyState rows yet). Health summary shows "not yet reported" for `healthy`.

### Host deleted between drawer open and operator interaction

The drawer was opened on an instance_uid that's since been deleted (e.g. another operator deleted the row, or magpied pruned a stale host). On any subsequent action attempt, the API returns 404. The drawer surfaces inline:

```
   ⚠ This host has been deleted from the registry.
   It may re-register on its next heartbeat with the same uid.
   
                                              [ Close drawer ]
```

The deep-link case (operator follows a stale URL) opens the drawer in this state directly.

### Permission failures

For v1.0 with single-shared-token, all-or-nothing: any 401 surfaces the existing sign-in modal (as v0.2). For v1.x with RBAC, individual actions might be denied (e.g. operator can read but not relabel) — out of scope for this spec.

### Concurrent edits

Two operators relabeling the same host concurrently: last-write-wins (current `PUT /agents/{uid}/labels` semantic). The drawer doesn't try to detect concurrent edits; it polls (2.5s) so an operator looking at the drawer sees the override change appear within 2.5s of the other operator's commit.

### YAML diff for "Clear override"

If the diff would be very large (entire config replaced), the diff renderer truncates to the first 50 changed lines + a "View full diff" link that expands inline. The confirm action is not blocked on diff readability — the operator can clear without viewing the full diff if they want.

## 13. Empty / loading / error states

### Loading

Drawer slides in immediately (no full-page spinner). Each section shows a small skeleton loader while data fetches. Health is the first to populate (cached in the agents poll); recent activity and in-flight rollouts populate ~50–200ms later (separate fetches).

### API errors during drawer

If a section's fetch fails:

```
   ⚠ Couldn't load recent activity for this host: <reason>
   
                                                  [ Retry ]
```

Inline within the section. Doesn't blank the drawer or block the rest.

### 404 on instance_uid (stale link)

```
   db-prod-7 was not found.
   
   This instance_uid may have been deleted, or the host may have
   never registered. If the agent is running it will re-register
   on its next heartbeat.
   
                                                [ Close ]
```

## 14. Accessibility

- Drawer trap-focuses on open; restores focus to the row that triggered it on close.
- Escape closes drawer.
- Headings hierarchy: `<h2>` for host name, `<h3>` per section ("Health," "In-flight rollouts," "Recent activity," etc.).
- Status dots have visible text counterparts ("healthy: yes," "drifting").
- Scope resolution chain is keyboard-navigable; the "in force" arrow announces via aria-label ("Currently in force: product+variant scope ship/linux").
- Live-updating regions (in-flight rollouts strip, recent activity) use `aria-live="polite"` so screen-reader users hear updates without being interrupted.
- All buttons are real `<button>` elements; "Open the config running here" is a button, not an anchor (it triggers a side-effect — opens another drawer — rather than navigating to a URL).

## 16. Test surface

### Component tests

- Drawer renders with default values for each diagnostic state (healthy / failed / drifting / pending / unhealthy).
- Scope resolution chain points at the right of three slots given `effective_product/variant` and override presence.
- "Open the config running here" button navigates to the right Product drawer.
- Re-assign cohort flow: empty → expand → fill inputs → submit → drawer reflects override-active state.
- Override config flow: opens publish dialog with `scope_kind=instance, scope_ref=<uid>` pre-set.
- Clear override: confirm dialog renders YAML diff; clicking confirm fires `DELETE` and clears the override-active state.
- In-flight rollouts strip: hidden at zero, visible at ≥ 1, max 3 cards + "+N more".
- Recent activity: 5 events shown, "View all" navigates to Audit section pre-filtered.
- Version drift banner appears only when versions differ.
- 404 from API surfaces the "not found" inline state without blanking the drawer.

### Integration tests (with magpied)

- Open drawer on a host → expected fields populate from `/agents/{uid}` + `/rollouts?state=in_flight` + `/audit?host_ref=<uid>`.
- Re-assign cohort → `PUT /agents/{uid}/labels` fires; drawer updates without manual refresh.
- Override config → publish dialog opens; dialog flow tested elsewhere (publish dialog spec).
- Clear override → `DELETE` against the override Rollout's instance Scope; drawer reflects the new resolution.
- Delete host → `DELETE /agents/{uid}`; drawer closes; Fleet view row disappears within poll cadence.

### Failure-mode tests

- API 5xx during drawer open → inline error per section, drawer remains usable.
- API 404 on initial load (stale link) → not-found state.
- Clear override fails (network blip) → inline retry, override remains active until success.
- Re-assign with empty product or variant → client-side rejection before submit.

### Accessibility tests

- Tab order: header close button → diagnostic action buttons → Scope chain → in-flight cards → re-assign expand → override expand → version drift download → all-attrs → delete.
- Screen reader announces the diagnostic line on drawer open (aria-live="assertive" — this is the load-bearing thing the operator came to read).
- Scope resolution chain announces "in force" position correctly.
- Confirm dialogs trap focus + restore on close.

## 17. Out of scope for this spec

- **Per-host metrics** (CPU, memory, log lines, etc.). Magpie isn't a telemetry store; the host drawer doesn't surface its agent's own metrics. Operators looking for those go to their OTLP backend (Datadog/Honeycomb/etc.).
- **Per-host policy / RBAC.** v1.x territory.
- **Host tagging / freeform notes.** v1.x — operators wanting to flag a host get the audit-log `payload_json` (per-action) but not a free-text note field on the host itself.
- **SSH-into-this-host / direct shell affordance.** Out of scope by deliberate design — Magpie is fleet-management, not remote-access.
- **Bulk operations spanning multiple host drawers.** Bulk happens in Fleet view; drawer is single-host.
- **Per-host log streaming.** Different product (cf. agent's collector exporters).
