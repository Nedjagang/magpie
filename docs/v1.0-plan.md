# Magpie v1.0 — System Redesign Plan

**Status:** In design. Decisions locked through 2026-05-09. Architecture spec'd, primitives defined, work items identified. **No code yet.**

**Version commitment:** v0.3 promoted to **v1.0** as a major-version release. Production-ready, source-of-truth for 2000+ agents. Designed at "Grafana-tier fleet management" rigor — architect-level discipline on primitives and invariants, no feature gold-plating beyond what reliability at scale requires.

**Threat model and operator audience:** Internal Aptean tool. Single trusted ops team (1–2 operators today, anticipated to grow). Internal-network deployment. Single-shared-token auth holds for v1.0; RBAC and OIDC are v1.x.

---

## 1. North star

Magpie v1.0 is the source of truth for OpenTelemetry collector configuration across 2000+ agents, with reliability strong enough that operators trust it as the *only* control plane for their fleet — not a tool they second-guess with SSH or backup config files.

A v1.0 publish to a 2000-host fleet must:

- Validate before any host sees the change.
- Reach a small canary first.
- Wait long enough to detect failure.
- Promote automatically when healthy, halt automatically when not.
- Be visible, pause-able, and reversible at every step.
- Leave a structured audit trail that survives tampering.

This is non-negotiable. Everything below exists to make it true.

---

## 2. Invariants

The design fails closed if any of these are violated. Anyone (operator, Claude, future contributor) proposing a change must satisfy all six or get explicit sign-off:

1. **Every state change is a Rollout** — identifiable, audit-able, reversible. No silent broadcasts, no implicit state mutations.
2. **Every host has exactly one effective config at any moment, and the system can show *which Scope produced it*.** Override > product > default. The resolution is explicit in the UI.
3. **The blast radius of any operator action is computed and shown before the click.** Aggregates at scale; exact counts always.
4. **Apply failures don't propagate beyond Canary.** A failing canary halts the rollout.
5. **Magpied is observable about itself.** `/metrics` endpoint is a v1.0 must-have.
6. **Agent autonomy survives control-plane failure.** Agents continue running last-applied config indefinitely if magpied is down. Telemetry never routes through Magpie.

## 3. Architecture — the five primitives

v1.0 introduces five first-class objects. Each replaces a v0.2 implicit concept that doesn't survive at 2000 scale.

### 3.1 Scope — addressable target for config

Three shapes. v1.0 ships the first two; the third is reserved in the schema:

- `product+variant` — v0.2's behavior. Default for most hosts.
- `instance_uid` — per-host override (Shape 1).
- `labels` — selector-based (Shape 2, **roadmap**).

**Resolution priority** when an agent connects: instance > product+variant > default. Magpied resolves Scope server-side from agent identity + advertised labels + operator overrides; the agent never picks. The resolution result is logged so the UI can show *exactly* why a host is on the config it's on.

**Why three shapes, not two:** Shape 2 requires the Scope abstraction to exist. Adding it now (even with one shape unimplemented) means the v1.x label engine doesn't require breaking changes.

### 3.2 Config — versioned artifact bound to a Scope

Fields:

- `id, scope_kind, scope_ref, name, yaml, hash, parent_revision_id`
- `validation_status: structural_ok | semantic_ok | invalid`
- `tags[]` — free-text human labels (e.g. `post-incident`, `q2-rollout`)
- `created_at, created_by`

**Validation runs both structural and semantic** — the actual OTel collector config loader runs before the row is accepted. A config that fails semantic validation cannot enter Canary.

One Config is **live** per Scope at any moment. Previous revisions are **available for rollback** but not active.

### 3.3 Rollout — keystone primitive

Every publish creates a Rollout. The Rollout is the unit of state change.

**Phases:**

| Phase | What happens | Gate |
|---|---|---|
| Validate | Structural + semantic check | Pass = advance; fail = aborted, no host sees the config |
| Canary | Config delivered to canary subset (default 5% or N=10, whichever is larger) | All canary hosts attempt apply |
| Soak | Watch ApplyState across canary, default 5 min | ≥ 95% applied successfully, no health regression |
| Promote | Config delivered to remaining hosts | Apply success aggregates |
| Done | Supersedes previous live Config (rollback-available) | — |

**Operator controls** at any phase:

- **Pause** — halt advancement, leave current state.
- **Abort** — roll back to previous live Config. Hosts on the new config get a re-push of the old.
- **Fast-Promote** — skip remaining gates, audit-warned. Used for hotfixes.

**Instant Rollout** — v0.2's 100%-in-2s semantic preserved as an explicit toggle in the publish dialog. Audit-typed distinctly from Phased rollouts (`RolloutInstant` vs `RolloutCreated`) so post-incident reviews can find them. **Default is Phased.**

### 3.4 ApplyState — per-agent, per-rollout state machine

States: `pending → applying → applied | failed`. Failed has `reason`, `attempt_count`. Stored, queryable, aggregable. A Rollout's health is the aggregate.

Powers the publish drawer's live status ("1840/2000 applied, 12 failed, 148 pending") and the Fleet's drift indicator (host's last `applied_config_hash` ≠ Scope's current `active_config_hash`, with no in-flight ApplyState explaining the gap).

### 3.5 AuditEvent — structured, hash-chained event

Schema: `id, prev_hash, hash, type, actor, scope_ref, config_ref, host_ref, timestamp, payload`.

**Event types:**

- *Rollout:* `RolloutCreated`, `RolloutAdvanced`, `RolloutPaused`, `RolloutAborted`, `RolloutPromoted`, `RolloutFastPromoted`, `RolloutInstant`
- *Override:* `OverrideApplied`, `OverrideCleared`
- *Host:* `LabelChanged`, `HostDeleted`
- *Auth:* `AuthSucceeded`, `AuthFailed` (rate-limited)

Hash chain (each entry's `prev_hash` matches the prior entry's `hash`) gives tamper-evidence with no external dependencies. Queryable by any entity ref + time range.

## 4. Information architecture

**Vocabulary:** "Product" stays. "Cohort" never appears in the UI. The org already speaks in product terms ("the ship product team").

**Sidebar (fixed, host-first):**

- **Fleet** — default landing. Hosts table, virtualized, sortable, filterable, multi-selectable. Status strip with rollups (healthy/total, in-flight rollouts, drifting hosts, failed applies).
- **Products** — `product+variant` Scopes with their live Configs and aggregate health. Authoring happens here.
- **Rollouts** — first-class. In-flight and completed rollouts with live phase progress. Drill in to abort/promote/inspect.
- **Audit** — first-class. Structured event stream with filters by entity + time + event type.

**Detail surfaces (right-side drawers):**

- *Host drawer* — leads with diagnostic, shows Scope resolution explicitly ("override / product+variant / default"), one-click cross-link to the live Config running on this host.
- *Product drawer* — leads with blast-radius header (host count + health roll-up), opens editor + Rollout history + Onboard flow.
- *Rollout drawer* — phase view, live ApplyState aggregates, controls (Pause / Abort / Fast-Promote).

## 5. Reliability guards (the UI enforces these)

These are not v1.0 *features*; they're constraints every screen must respect.

- **Blast radius shown before every write/destroy.** Aggregates with breakdown at 2000 scale ("1840 hosts in 3 products, 12 with overrides drifting"); exact counts always.
- **Cross-product bulk-edit-config blocked.** Multi-select spanning multiple Products greys out "Edit config" with explanation. Bulk relabel and bulk delete *are* allowed across Products (they don't violate Scope semantics).
- **Bulk delete confirms show top-N rows + count.** Full list available on demand.
- **Override clear shows YAML diff** before applying.
- **Healthy-host delete warns, doesn't block.** Agents re-register on heartbeat — surface this explicitly.
- **Live rollout status in publish drawer.** No navigation away after Publish.
- **Instant Rollout requires explicit toggle.** Default is Phased. Audit-typed distinctly.

## 6. Work breakdown

Items required for v1.0 ship, grouped by track. Order within a track is a dependency hint, not a release schedule.

### 6.1 Backend / engine

1. **Postgres backend option.** Pull the trigger on ADR 0007. Add as alternative to SQLite. Migrations runnable on either. Connection-pool config. **✓ wired and validated end-to-end (`internal/db/`): pgx/v5/stdlib driver, `Open(dsn)` returns `*db.Conn` (wraps `*sql.DB` + dialect, auto-Rebinds `?` ↔ `$N` via `Exec`/`Query`/`QueryRow`). 9 migrations ported to `migrations/postgres/` next to the existing `migrations/sqlite/` set. Stores (agents/audit/configs/labels) updated to take `*db.Conn`. `configs.Create` uses `INSERT … RETURNING` for portable id round-trip. End-to-end validation via `internal/db/postgres_test.go` (build-tagged `pg`, embedded-postgres) — runs all 9 migrations against PostgreSQL 18.3, exercises every store roundtrip (configs Create/Get/List/LatestFor/pagination, agents Upsert ON CONFLICT + three-state Healthy via SMALLINT, audit Record + List, labels Set/Get/Clear) plus the append-only triggers (UPDATE/DELETE both raise `audit_log is append-only`). `make test-pg` runs the suite.**
2. **Pagination** on `/api/v1/agents`, `/api/v1/configs`, `/api/v1/audit`. Cursor-based. Default page size sane for 2000-host fleet. **✓ landed (`internal/pagination`): `?limit=` (default 500 / 100 for audit, max 5000), `?cursor=` (opaque base64), `X-Next-Cursor` + `X-Total-Count` response headers. Raw-array response shape preserved so the v0.2 UI keeps working; v1.0 Fleet view spec teaches the UI to follow cursors.**
3. **Scope primitive** as a first-class concept in the data model. `configs` table grows `scope_kind` and `scope_ref` columns. v1.0 supports `product_variant` and `instance` kinds; `labels` reserved. **✓ schema landed (migration `00007_scope_columns.sql`); Go layer pending.**
4. **Per-host override engine (Shape 1).** Resolution order: instance > product+variant > default. Operator applies overrides via existing label-override UI extended for per-host config selection.
5. **Rollout primitive.** New `rollouts` table. State machine implementation. Phase advancement. Gate evaluation. Pause / Abort / Fast-Promote endpoints. **✓ end-to-end (`internal/rollouts/` + opamp + main.go ticker): Create flow synchronously runs through Validate → Canary, OpAMP push pipeline now consults `rolloutSvc.ResolveConfigFor` (in-flight rollouts → instance Scope live → product+variant Scope live → v0.2 fallback), `Service.ResolveConfigFor` side-effect transitions apply_state pending → applying, agent heartbeats route through `Server.SetHeartbeatHook` → `Service.HandleHeartbeat` to transition applying → applied|failed. Background ticker (`runRolloutTicker` in main.go, 10s cadence) calls `AdvancePhase` for every non-terminal rollout, walking canary → soak → promoting → done with real soak-window timing (Clock interface, system clock in production, fake clock in tests). Auto-abort on canary gate failure honored. 25 tests in the rollouts package, all green.**
6. **ApplyState primitive.** New `apply_state` table per (host, rollout). State machine driven by OpAMP heartbeat reports. **✓ end-to-end via `Service.HandleHeartbeat` invoked from `opamp.Server`'s `HeartbeatHook`. Transitions on hash match between agent's reported applied_config_hash and the rollout's target Config hash; configFailed=true marks failed with operator-readable error.**
7. **Semantic config validation.** Run OTel collector config loader before accepting a publish. Reject on failure with a clear error. **✓ landed as opt-in (`internal/semantic/`, design at [`docs/v1.0-semantic-validation.md`](./v1.0-semantic-validation.md)): subprocess `otelcol-contrib validate --config=<temp>` with 5 s timeout, stderr captured + truncated to 8 KB, surfaced to the publish dialog via `ErrValidate`. Distinct abort reason `semantic_validate_failed`. **2026-05-11 update:** the default posture switched from "auto-discover on $PATH" to **opt-in via explicit `MAGPIE_OTELCOL_VALIDATOR_PATH`**. Rationale: the validator is platform-bound (a Windows binary can't validate Linux-only `journald`), and the agent's own otelcol catches bad configs at apply time and reports via ApplyState — so publish-side gating, while useful, isn't load-bearing for reliability. Operators who want the gate set the env var; everyone else runs structural-only and trusts otelcol semantics. The `$PATH` auto-fallback was removed because it produced false negatives when the magpied host's binary diverged from the fleet's.**
8. **Hash-chained audit log.** Schema migration, hash computation on insert, verification endpoint. **✓ Go layer landed (`internal/audit/`): `EventType` enum (12 types per spec §11), `Event` struct, `Store.RecordEvent` with hash-chain (mutex-serialized; SHA-256 over canonical pipe-form of prev_hash | type | actor | scope_kind | scope_ref | config_ref | host_ref | payload_json | at), `Store.VerifyChain` returns (validUpToID, brokenAt, err) — chain validator detects out-of-band UPDATEs even when audit_log triggers are bypassed. `Store.Record` retained as v0.2 compat (no chain). All `rollouts.Service.recordAudit` call sites migrated to RecordEvent with structured payloads. New endpoint `GET /api/v1/audit/verify`. 3 new tests: chain integrity, tamper detection, legacy+chain coexistence.**
9. **`/metrics` Prometheus endpoint.** Connected agents, push queue depth, rollout phase counts, validation latency, db query duration, apply-state aggregates. **✓ rollout-aware drop landed: `magpie_info`, `magpie_connected_agents`, `magpie_http_requests_total`, `magpie_http_request_duration_seconds`, `magpie_apply_state_total{state}`, `magpie_rollout_phase_active{phase}`, `magpie_validation_latency_seconds`. Rollout-aware gauges are scrape-time-pull (custom `prometheus.Collector` impl in `internal/metrics`); `RolloutAggregator` interface implemented by `rollouts.Store`. Validation histogram observed inline via `ValidationObserver` interface satisfied by `metrics.Registry`. `db_query_duration_seconds` deferred (independent track).**
10. **Drift detection.** Periodic (or on-event) computation of per-host drift; surface in agent rows.

## 9. Out of scope for v1.0

Each of these is real work that's been considered and **explicitly deferred**. Reopening any requires re-justifying against the v1.0 must-have list and updating this document.

- **HA / multi-magpied with shared Postgres** — v1.1. Postgres backend enables it; HA itself is the next increment.
- **Selectors + arbitrary host labels (Shape 2)** — designed for; not built in v1.0.
- **RBAC scoped tokens + OIDC for operators** — v1.x. Internal-network single-team threat model holds for v1.0.
- **`magpiectl` CLI + OpenAPI spec stability promise** — v1.x. Needed for GitOps/CI, not for the v1.0 reliability bar.
- **GitOps reconcile mode** — v1.x.
- **Per-host canary selection** — v1.0 canary is %-based or N-based. "Pick these specific hosts as canary" is v1.x.
