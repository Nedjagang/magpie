# Fleet view — design spec (v1.0)

**Status:** Design only. **No code yet.** Open questions in §14 must be confirmed (or working assumptions accepted) before implementation begins.

**Scope of this document:** the **Fleet** sidebar entry — the default-landing host-first IA that operators open first. Covers status strip, in-flight rollouts strip, filter bar, virtualized host table, multi-select + bulk actions, cross-product Edit-config guard, deep-linking, performance/scale budgets, and accessibility. Does **not** cover the host drawer (separate spec), product drawer (existing v0.2 affordance), publish dialog (separate spec), or audit query surface (separate spec).

**Reading order:** §1–§3 frame why this surface matters and how it's structured; §4–§8 walk through each pane; §9–§11 cover bulk operations + cross-product semantics + state surfaces; §12 covers performance at 2000 scale; §13–§16 cover accessibility, open questions, test surface, deferrals.

**Supersedes / complements:** [`docs/v1.0-plan.md`](./v1.0-plan.md), [`docs/v1.0-rollout-spec.md`](./v1.0-rollout-spec.md), [`docs/v1.0-publish-dialog-spec.md`](./v1.0-publish-dialog-spec.md). On any conflict the plan wins for scope/cut-line questions; the relevant spec wins for its surface; this spec wins for Fleet-view UX.

---

## 1. Why this surface matters

Magpie v1.0 made the IA host-first (plan §4): the operator lands on the fleet, drills into a host to fix things, drills into a product to author things. Fleet is the default landing — the first thing every operator sees, every session. At 2000-host scale, this surface has to do four jobs *at the same time*:

1. **Tell me if anything's on fire** in under 1 second of page load (status strip).
2. **Let me find a specific host fast** when I have a hostname or a vague description ("that drifting one in pay/linux") (filter + search).
3. **Let me operate on a group of hosts** safely (multi-select + bulk actions, with blast radius surfaced before every click).
4. **Stay responsive** at 2000 rows, sorting, filtering, scrolling, polling — without making the operator wait or watch the browser stutter.

If any one of those fails, the rest of the IA's host-first promise unravels. An operator who can't find what they need in Fleet will SSH instead, and Magpie stops being source of truth.

---

## 2. Where the operator comes from

Three entry paths:

1. **Default landing.** Open `magpie.example.com/`. Sidebar selects Fleet automatically. Filters at default (none). Sort at default (last-seen DESC).
2. **Sidebar click.** From any other view (Products / Rollouts / Audit). Fleet view restores the operator's last filter+sort state for the session (open question §14 #5: persist across sessions or session-only).
3. **Deep link.** A URL like `magpie.example.com/fleet?status=failed&product=ship` opens Fleet pre-filtered. The full filter state is URL-serializable (§11) so operators can share a link to a slice of the fleet — e.g. "all failed-apply hosts on ship" — without describing it in prose.
4. **Status strip click-through.** From within Fleet itself, clicking a metric in the status strip is a filter shortcut — the URL updates so the resulting filtered view is shareable.

## 3. Structural anatomy

The Fleet view is a single column with a fixed-height status region on top and a flexible host table below. Schematically:

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  Fleet                                                  [ Onboard a host ▸ ] │
├──────────────────────────────────────────────────────────────────────────────┤
│  ┌────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌───────┐ │
│  │ 1,847 hosts│ │ 1,830 healthy│ │ 17 unhealthy │ │ 23 drifting  │ │ 4 ✗   │ │
│  │ total      │ │   ●          │ │   ●          │ │   ●          │ │ failed│ │
│  └────────────┘ └──────────────┘ └──────────────┘ └──────────────┘ └───────┘ │
├──────────────────────────────────────────────────────────────────────────────┤
│  In-flight rollouts (2)                                                      │
│  ┌────────────────────────────┐  ┌────────────────────────────┐              │
│  │ ship/linux       ▸ Soak    │  │ pay/windows      ▸ Promote │              │
│  │ 4m 12s remaining            │  │ 318/320 applied             │              │
│  │ alice@aptean · 2m ago       │  │ bob@aptean · 6m ago         │              │
│  └────────────────────────────┘  └────────────────────────────┘              │
├──────────────────────────────────────────────────────────────────────────────┤
│  ┌─[ search hostnames... ]─┐  Status: [Any▾]  Cohort: [Any▾]  Seen: [24h▾]  │
│  └─────────────────────────┘                                  [ Clear all ] │
├──────────────────────────────────────────────────────────────────────────────┤
│  ☐ │ Host                Cohort           Status   Applied    Last seen      │
│  ──┼─────────────────────────────────────────────────────────────────────── │
│  ☐ │ ● db-prod-7        ship/linux       failed   3a91…       12s ago       │
│  ☐ │ ● db-prod-3        ship/linux       applied  7c02…       18s ago       │
│  ☐ │ ● vm-staging-12    pay/windows      applied  9f02…       22s ago       │
│  ☐ │ ● ...                                                                  │
│  ──┴─────────────────────────────────────────────────────────────────────── │
│  Showing 50 of 1,847                                       [ Load more ▸ ]  │
└──────────────────────────────────────────────────────────────────────────────┘
```

Five structural zones, each independently rendered:

- **Header row.** Page title + primary CTA (Onboard a host).
- **Status strip.** Five metric cards. Always visible; each clickable.
- **In-flight rollouts strip.** Conditionally visible (hidden when zero rollouts in flight). Horizontal cards, scroll right if many.
- **Filter bar.** Search box, three filter chips, clear-all link.
- **Host table.** Virtualized scroll, multi-select column, sortable headers, per-row drilldown.

Below the table, a "Showing N of M" line and a "Load more" affordance for cursor pagination (§7.5).

## 4. Status strip

Five metrics, each a card. Three reasons they're the *first* thing an operator sees:

1. **Glance test.** An operator opening Fleet to verify "anything on fire?" should answer that question without reading more than the strip.
2. **Filter shortcut.** Each metric is clickable; clicking applies the corresponding filter and updates the URL.
3. **Live update.** The strip polls every 2.5s (matches v0.2 cadence) so an operator who keeps the tab open watches the fleet evolve.

### The five metrics

| Metric | Source | Click filters to |
|---|---|---|
| **N hosts total** | `len(GET /api/v1/agents)` aggregate (`X-Total-Count` header) | no-op (already shows all) |
| **N healthy** | `agents` filtered to `healthy=true` | `?status=healthy` |
| **N unhealthy** | `agents` filtered to `healthy=false` | `?status=unhealthy` |
| **N drifting** | `agents` whose `applied_config_hash` ≠ live Scope hash + no in-flight ApplyState | `?status=drifting` |
| **N failed apply** | `agents` whose `config_status=failed` | `?status=failed` |

### Visual treatment

- **Total**: neutral (no dot indicator). It's a sum, not a state.
- **Healthy**: small green dot, deemphasized (this is the boring number).
- **Unhealthy**: small yellow dot if N > 0, neutral if N = 0.
- **Drifting**: small yellow dot if N > 0, neutral if N = 0.
- **Failed**: small red dot if N > 0, neutral if N = 0. Most attention-grabbing.

Cards remain visible at zero. Operators benefit from seeing "0 failed" — that's the all-clear signal.

### What's deliberately not in the strip

- **Connected vs total** distinction. v0.2 had this; it confused operators ("I see 1840 hosts but 23 'unhealthy' — does the count include disconnected ones?"). v1.0 collapses this: hosts in the table are connected hosts; healthy/unhealthy/drifting/failed are sub-states of connected. Deleted-and-pruned hosts don't appear (host deletion in v0.2 leaves a row that re-registers; v1.0 keeps that, but the row's `last_seen` updating is the freshness signal, not a separate "connected" metric).
- **Per-product breakdown.** Operators who want it can click into Products section. Strip is fleet-wide.

## 5. In-flight rollouts strip

Conditionally visible. Hidden when zero rollouts are non-terminal. When ≥ 1, a horizontal scrollable strip of compact rollout cards appears below the status strip.

### Card content

Each card shows:

```
┌────────────────────────────┐
│ ship/linux       ▸ Soak    │
│ 4m 12s remaining           │
│ alice@aptean · 2m ago      │
└────────────────────────────┘
```

- **Top row.** Scope ref + current phase (with one of the standard pill colors per phase).
- **Middle row.** Phase-specific progress: countdown for Soak, applied/total for Canary or Promote, "validating…" for Validate.
- **Bottom row.** Operator + age.

Click → opens the **Rollouts section drawer** for that rollout (separate spec). The publish dialog's "View in-flight rollout" link lands here too.

### Behavior at scale

- 1–4 rollouts: cards displayed inline, no scroll.
- 5+ rollouts: horizontal scroll within the strip. Visible-card count adjusts to viewport width.
- 20+ rollouts (very rare): a "View all 23 in-flight rollouts" link replaces the strip — listing each as a card would dominate the page.

### What's deliberately not here

- Recently-completed rollouts. The strip is *in-flight only* — it's a status-of-active-things surface, not a rollout history. History lives in the Rollouts section.
- Inline operator controls (Pause/Abort/Promote/Fast-Promote on the card). The card is summary; controls live in the dedicated rollout drawer to keep the Fleet view focused.

## 6. Filter bar

Three filter chips + a search box + clear-all.

### Search

Plain text input, debounced 200ms. Matches against host name (`attributes["host.name"]`) and `instance_uid`. Case-insensitive substring match.

At 2000 hosts, even client-side search is fast enough to not need a server round-trip — the data is already loaded for the visible page. (Open question §14 #6: server-side search across paginated results vs client-side within current page.)

### Status chip

Single-select dropdown:
- Any (default)
- Healthy
- Unhealthy
- Drifting
- Failed apply

(Open question §14 #7: support multi-select status, e.g. "unhealthy OR failed"?)

### Cohort chip

Two-stage select. Click → menu appears with two columns: Product (left), Variant (right). Selecting a Product narrows the Variant column. Both can be "Any."

### Last-seen chip

Single-select dropdown:
- Any (default)
- 1 hour
- 24 hours (default working-set view)
- 7 days
- 30 days

Hosts beyond this window are filtered out — useful for excluding decommissioned-but-not-yet-pruned rows.

### Clear all

Resets every filter chip + search box. Sort and column visibility unchanged.

## 7. Host table

The main surface. Sortable, filterable, virtualized, clickable rows, multi-selectable.

### 7.1 Columns

Default columns, in order:

| Column | Source | Default sort |
|---|---|---|
| Selection checkbox | (UI state) | — |
| Host indicator (●) + name | `attributes["host.name"]` | A→Z within ties |
| Cohort | effective `(product, variant)` | — |
| Status | derived from `healthy`, `config_status`, drift | — |
| Applied | first 8 chars of `applied_config_hash` | — |
| Last seen | `last_seen` | **DESC** (default) |

Status column shows one of:
- `● applied` (green dot — healthy + config matches scope)
- `● drifting` (yellow dot — applied hash differs from scope's live)
- `● failed` (red dot — `config_status=failed`)
- `● unhealthy` (red dot — `healthy=false`)
- `● applying` (spinner — has in-flight ApplyState row)
- `● pending` (outline dot — agent registered but hasn't reported yet)

Applied column shows the agent's `applied_config_hash` (first 8 chars in monospace) plus a small drift indicator if it differs from the scope's live hash:
- `7c02ab94` (matches live — no decoration)
- `7c02ab94 ↗` (drifting — small icon, hover shows "live: 9f02…, drift since 2h ago")

### 7.2 Sorting

Click any header to sort. Click again to reverse. Default is last-seen DESC.

Sort applies to the *currently loaded* page set (cursor pagination). When the operator sorts and then loads more, the next page's cursor follows the new sort key. (Open question §14 #4: server must support sort by every column for this to work — confirm /agents API exposes sort params; if not, we add to the cursor pagination work.)

### 7.3 Density

Comfortable spacing by default — every row has room for a status dot, primary text, secondary text, monospace hash. A toggle in the table header switches to **Compact** (denser rows, secondary text trimmed) for operators who want to see more rows on screen.

Compact mode is per-session (open question §14 #5).

### 7.4 Row click

Click anywhere outside the checkbox column → opens **host drawer** (separate spec).

### 7.5 Pagination

The Fleet view uses cursor-based pagination from the `/api/v1/agents` API (already landed). Default page size 500. The "Showing N of M" line and "Load more" link appear under the table:

```
Showing 500 of 1,847                                      [ Load more ▸ ]
```

"Load more" appends the next 500 to the visible set. Operators who want to see-all-then-bulk-act can repeatedly click; or they can use the multi-select "select all matching filter" affordance (§9) to select across pages without loading every row into the DOM.

Infinite scroll is not used — it's harder to keep accessibility ARIA-correct, and "Load more" gives a clear stopping point that operators can interrupt (close the tab, refile the filter) without losing work.

### 7.6 Virtualization

Even within a 500-row page, full DOM rendering causes layout cost when the operator scrolls fast. The table uses a windowing library (react-window or react-virtual; choice is implementation detail) that renders only visible rows + a small overscan buffer.

Performance budget: scroll a 500-row table at 60fps on a 4-core laptop without main-thread jank. Validated by the test surface in §15.

## 8. Multi-select + bulk action bar

The leftmost column is a per-row checkbox. The header has a **selection-mode** checkbox.

### 8.1 Selection states

Three meaningful selections:

1. **Visible** — the rows currently in the operator's viewport (~50 in a default scroll position).
2. **Loaded** — the rows currently fetched (e.g. 500 if "Load more" hasn't been clicked yet).
3. **All matching filter** — every host the current filter set would match across all pages (e.g. 1,840 if the operator is browsing healthy ship/linux hosts).

The checkbox in the header initially toggles **visible**. After it's checked, a banner appears:

```
   ┌────────────────────────────────────────────────────────────┐
   │ 50 hosts selected.   [ Select all 1,840 matching filter ]  │
   └────────────────────────────────────────────────────────────┘
```

Clicking the link upgrades the selection to "all matching filter." This pattern (familiar from GitHub) makes the intent explicit — operators can't accidentally bulk-act on 1,840 hosts when they meant 50.

### 8.2 The action bar

When ≥ 1 host is selected, a sticky action bar appears at the top of the table (replacing the filter bar's space, or overlaying it; visual choice is implementation detail).

```
   ┌────────────────────────────────────────────────────────────────┐
   │ 5 hosts selected.   [ Re-assign cohort ]   [ Edit config ]     │
   │                     [ Delete ]              [ Cancel ]          │
   └────────────────────────────────────────────────────────────────┘
```

Three actions:

1. **Re-assign cohort.** Opens a modal that asks "which cohort?" then writes label overrides for every selected host. Per-host writes; the action bar shows progress ("Re-assigning 5 hosts… 3 of 5 done"). Allowed across products (per plan §5 reliability guards: bulk relabel is allowed across products; bulk-edit-config is not).
2. **Edit config.** Special handling — see §9.
3. **Delete.** Bulk delete with full confirm. Confirm enumerates rows (top-N + count, per spec §8.3).

### 8.3 Bulk delete confirm

The confirm modal:

```
   Delete 5 host records?
   
     vm-old-1     ship/linux       last seen 8d ago
     vm-old-2     ship/linux       last seen 8d ago
     vm-old-3     pay/linux        last seen 2 minutes ago     ⚠ recently healthy
     vm-old-4     pay/windows      last seen 8d ago
     vm-old-5     default/default  last seen 8d ago
   
   ⚠ vm-old-3 is currently healthy. Deleting will let its agent
     re-register on its next heartbeat.
   
                                       [ Cancel ]   [ Delete 5 hosts ]
```

At larger scale (50+ rows), the confirm enumerates the first 20 + count:

```
   Delete 1,840 host records across 3 products?
   
     ship/linux × 1,200
     ship/windows × 320
     pay/linux × 320
   
   First 20 to be deleted:
     vm-old-1 (ship/linux, 8d ago)
     vm-old-2 (ship/linux, 8d ago)
     ...
   [ View all 1,840 ▸ ]
   
   ⚠ 3 of these are currently healthy.
   
                                       [ Cancel ]   [ Delete 1,840 hosts ]
```

The "View all" link expands to a virtualized list within the modal — operators can scan all 1,840 if they want. Default-collapsed because reading 1,840 entries before clicking is unrealistic.

The confirm button names the count explicitly: **"Delete 1,840 hosts"** rather than just "Delete." This is the load-bearing reliability beat — operators see what they're committing to.

### 8.4 Operation execution

Bulk delete fan-out uses the existing `DELETE /api/v1/agents/{uid}` endpoint per host. The action bar shows progress and surfaces individual failures inline:

```
   Deleting 1,840 hosts...   1,820 deleted · 18 failed · 2 in progress
   
   ⚠ Some deletes failed:
     vm-prod-93     network error (will retry)
     vm-prod-127    not found (already deleted)
     ... +16 more
   
   [ Retry failures ]   [ View errors ]
```

Operators can retry the failures or close out. Successful deletes are persisted regardless of partial failure.

(Open question §14 #8: bulk endpoint that takes an array of UIDs vs N parallel calls. v1.0 starts with N parallel calls (no API change needed); a `POST /api/v1/agents:bulkDelete` may be worth adding in v1.x if performance demands.)

## 9. Cross-product Edit-config guard

Per plan §5 and Rollout spec §3 concurrency rule, **bulk Edit config across multiple products is impossible** because publish is Scope-scoped — a single Rollout can only target one Scope.

### 9.1 When it triggers

The "Edit config" button in the bulk action bar is enabled only when **all selected hosts share the same effective `(product, variant)` cohort**. Three cases:

1. **Selection in one cohort** (e.g. all 5 selected are ship/linux). Button enabled. Click → opens publish dialog targeting ship/linux.
2. **Selection spans multiple cohorts in one product** (e.g. 5 ship/linux + 3 ship/windows). Button enabled-with-warning state. Click → modal asks the operator to pick one cohort to publish to:

```
   These 8 hosts span 2 cohorts of "ship":
     ship/linux  × 5
     ship/windows × 3
   
   Publishing requires picking one cohort. The config goes to all
   hosts in that cohort, not just your selected ones.
   
   ◯ ship/linux (12 hosts total — your 5 selected + 7 others)
   ◯ ship/windows (4 hosts total — your 3 selected + 1 other)
   
                                       [ Cancel ]   [ Continue ▸ ]
```

The modal makes the cohort-write-semantic explicit: "your 5 + 7 others." The operator picks one and continues to the publish dialog. The "+ N others" disclosure prevents the "I selected 5 but it went to 12" surprise.

3. **Selection spans multiple products** (e.g. ship/linux + pay/linux). Button **disabled** with tooltip:

```
   Edit config disabled
   
   These 8 hosts span 2 products: ship, pay. Magpie publishes to a
   single (product, variant) at a time. Either narrow your selection
   to one product, or open each product's publish dialog separately.
```

### 9.2 Why "disabled with tooltip" rather than a modal

For multi-product, the right operator action is "go to Products section and publish per product." A modal trying to walk them through that workflow is friction; a clear tooltip + sidebar redirect is enough.

## 10. Empty / loading / error states

### Empty fleet

The first time magpied boots and no agent has connected, Fleet shows:

```
   No hosts yet.
   
   Onboard your first host with the [ Onboard a host ▸ ] button above,
   or check the docs/onboarding.md for manual installation.
```

### Filter returns zero

```
   No hosts match the current filter.
   
   [ Clear filters ]
```

### API error

If `/api/v1/agents` returns 5xx or the network is unreachable, the host table shows:

```
   ⚠ Couldn't reach magpied: <reason>
   
   [ Retry ]
```

Status strip and rollouts strip are preserved if their last-good data is < 30s old (don't blank the page on a transient network blip). Older than 30s, they show their own "last updated 2m ago" stale indicator.

### Auth expired (401)

The sign-in modal from v0.2 carries forward — no Fleet-specific handling.

## 11. Deep linking / URL state

Filter and sort state is encoded as query params:

| Param | Values |
|---|---|
| `status` | `healthy | unhealthy | drifting | failed | applying | pending` |
| `product` | any string |
| `variant` | any string |
| `seen` | `1h | 24h | 7d | 30d` |
| `q` | search string (URL-encoded) |
| `sort` | `last_seen | name | cohort | status | applied` |
| `dir` | `asc | desc` |

Cursor pagination is **not** in the URL — cursors are opaque + ephemeral, and a stale link is worse than a fresh one starting at the top. "Showing N of M" makes the position visible enough.

URL state syncs both ways: filter chip change updates URL; URL change (back/forward, manual edit) updates filter chips.

Examples:
- `/fleet?status=failed` — all failed hosts.
- `/fleet?product=ship&seen=1h` — ship product, active in last hour.
- `/fleet?q=db-prod&sort=last_seen&dir=desc` — search + sort.

### URL hygiene

Default values are *omitted* from the URL — `?seen=24h` (the default) doesn't appear. Only non-default values do. Keeps shared links short and meaningful.

## 12. Performance + 2000-scale

### Rendering budget

- **Time to interactive on first paint** (cold): < 2 seconds at 2000 hosts. The first 500 host rows are visible; the rest load via "Load more."
- **Scroll FPS**: ≥ 55 fps on a 4-core laptop with 500 rows in the virtualized table.
- **Filter chip change**: the filter applies to the loaded set in < 100ms. Server-side re-fetch (when the filter requires it) finishes in < 500ms at p95.

### Polling

- Status strip + rollouts strip + table data: 2.5s base interval (matches v0.2).
- **Background tab backoff**: when the tab is not visible (Page Visibility API), polling backs off to 30s. Operators alt-tabbing back get a fresh refresh, but Magpie isn't burning bandwidth on hidden tabs.

### Data flow

The Fleet view fetches three things on mount:
- `GET /api/v1/agents?limit=500` (the host list).
- `GET /api/v1/agents` for status strip totals — but the existing endpoint returns the array; we get totals from the `X-Total-Count` header. (Open question §14 #1: is this enough, or do we need a dedicated `/api/v1/agents/stats` for the rollups?)
- `GET /api/v1/rollouts?state=in_flight` (for the rollouts strip).

Subsequent polls hit the same endpoints. Cursor-paginated "Load more" appends to the host list without re-fetching the whole thing.

### Cardinality of the filter set

The status-chip filters are derived; the cohort chip is dynamic (lists products + variants present in the current fleet). At 2000 hosts spread across, say, 10 products and 30 variants, the cohort dropdown has ~30 entries. Rendered as a virtual list inside the dropdown if it grows past 100 (open question §14 #2: cap it, paginate it, or search-within-dropdown).

## 13. Accessibility

- Table uses semantic `<table>` markup (not divs) so screen readers announce rows + columns correctly.
- Sort buttons in headers have `aria-sort="ascending|descending|none"`.
- Status dots have visible text counterparts ("healthy", "failed") not just color.
- Selection checkbox column has clear `aria-label` per row ("Select db-prod-7").
- Live-updating regions (status strip metrics, rollouts strip cards) use `aria-live="polite"` so screen-reader users hear updates without being interrupted.
- Filter bar inputs are keyboard-navigable; focus order: search → status → cohort → seen → clear all → first row checkbox → first row.
- The "Load more" button is a real button (not a div with onclick) — Enter/Space activates.
- Confirm dialogs trap focus and restore on close (standard modal pattern).

## 14. Open questions (working assumptions to confirm before code)

> **Status (2026-05-09):** all working assumptions in this section are **committed** in [`docs/v1.0-decisions.md`](./v1.0-decisions.md). Treat the table below as decided unless the decisions doc has been overridden.

| # | Question | Working assumption |
|---|---|---|
| 1 | Aggregate stats endpoint vs `X-Total-Count` header | Use `X-Total-Count` for v1.0; consider `/api/v1/agents/stats` if rollups need to be cheaper than walking the whole set |
| 2 | Cohort dropdown cap behavior | Cap visible at 100, search-within-dropdown for the rest |
| 3 | Healthy host delete warning threshold | "Recently healthy" = `last_seen < 5 min ago` |
| 4 | Sort support per column on `/agents` API | Confirm API supports `?sort=...` params; if not, add to pagination work |
| 5 | Filter+sort state persistence | Session-only for v1.0; per-user persistence is v1.x |
| 6 | Search: server-side vs client-side | Client-side within loaded set for v1.0; server-side `?q=` is v1.x if a 500-row page isn't enough |
| 7 | Multi-select status filter | Single-select for v1.0; multi-select is v1.x |
| 8 | Bulk delete: N parallel calls vs `:bulkDelete` endpoint | N parallel for v1.0; `:bulkDelete` if perf demands |
| 9 | Density toggle (compact/comfortable) | Per-session, defaults to comfortable |
| 10 | "View all 1,840" expanding modal | Virtualized list inside modal, capped at 5,000 rows (above that we render a count + "save filter and review in chunks" path; v1.x) |
| 11 | Drift detection lag | Surface a "computed N seconds ago" tooltip on the drifting count if older than 30s |
| 12 | `last_seen` rounding | "12s ago" / "2m ago" / "8d ago" relative; absolute on hover |

## 15. Test surface

### Component tests

- Status strip renders five cards; clicking each updates the URL with the corresponding filter.
- In-flight rollouts strip is hidden when zero; visible when ≥ 1; horizontally scrollable past 4.
- Filter chip changes update URL (and vice versa).
- Sort header click toggles ascending/descending; second click reverses.
- Selection: header checkbox selects visible; "select all matching filter" link upgrades.
- Cross-product Edit config: 1 cohort → enabled; 2 cohorts in 1 product → modal-pick; 2 products → disabled with tooltip.

### Integration tests (with magpied)

- Initial load fetches `/agents`, `/rollouts?state=in_flight`, populates strip + table.
- "Load more" follows `X-Next-Cursor` header.
- Polling cadence: 2.5s foreground, 30s background.
- Bulk delete: 5 selected → 5 `DELETE /agents/{uid}` calls; partial-failure surfaces inline retries.
- Re-assign cohort: 5 selected → 5 `PUT /agents/{uid}/labels` calls.

### Performance tests

- 500-row table scrolls at ≥ 55 fps (Lighthouse / Chrome perf trace).
- Filter chip change applies in < 100ms (loaded set) on a 500-row page.
- Cold first paint < 2s at 2000 hosts (using staged data — first 500 visible, rest behind Load more).

### Failure-mode tests

- API 5xx during initial load → error state with retry.
- API 5xx during poll → keep last-good data ≤ 30s, then show stale indicator.
- Selection across pages: select 50 visible, click "select all 1,840", verify bulk delete confirmation accurately enumerates by cohort.
- 401 during poll → sign-in modal (existing v0.2 behavior).

### Accessibility tests

- Tab order matches §13.
- Screen reader announces sort order changes + filter chip changes.
- All status pills have visible text counterparts.
- Confirm modal traps focus + restores on close.
