# v1.0 working-assumption decisions

**Status:** Decided 2026-05-09. Every "working assumption" called out in the four v1.0 specs is committed here, with alternatives considered and the reasoning that picked the working assumption. **Reading the specs going forward, treat anything in their Open Questions sections as decided per this document** unless you find a later override.

**Reversal is possible** but not silent. To revisit a decision: find the entry below, update both this document and the corresponding spec section, and note the date. The specs are design records; this document is the rolling registry of what's actually committed.

**On the bias of these decisions:** in the absence of operator pushback, every assumption resolves toward (a) the safer default at scale, (b) the smaller v1.0 surface, (c) the path that preserves the reliability invariants from the plan §2. Where a decision is genuinely consequential (changes architecture, changes operator behavior at scale), it's flagged ⚠ — those are the ones to push back on if any of them feel wrong.

## Rollout primitive ([`v1.0-rollout-spec.md`](./v1.0-rollout-spec.md) §13)

### 1. Default canary size

**Decision:** `max(ceil(5% × connected_hosts), 10)` — whichever is larger.
**Why:** the `N=10` floor protects small fleets (a 5% canary at 50 hosts = 3 hosts, too thin to detect a regression statistically). The `5%` cap protects large fleets (10 of 2000 hosts is a 0.5% canary, too thin to be meaningful).
**Alternative:** per-product configurable defaults (operator can pin `ship/linux` at canary=20). Deferred to v1.x — the global default works for "everyone" without giving operators yet another knob to set.

### 2. Default soak window

**Decision:** 300 seconds (5 minutes).
**Why:** enough time for an OpAMP push + apply + collector self-checks + agent heartbeat reporting back to magpied. Short enough that operators don't time out.
**Alternative:** 60s (too tight — apply failures don't always surface in 60s on a host with a slow disk or contended scheduler) or 15min (operator boredom). 5 min sits in the sweet spot.

### 3. Auto-rollback on canary gate failure ⚠

**Decision:** **default-on, no opt-out in v1.0**.
**Why:** failure is bounded to canary subset (≤ 5% / N=10) — auto-aborting a contained failure is the move that loses *least* on the operator's behalf. Leaving a failing canary in `soak` waiting for a human operator gives more time for failures to compound (agent restart loops, retry storms).
**Override path:** operator can hit Pause as soon as gate evaluation reports failed (UI updates every 10s; auto-abort fires on the *next* cycle, not the same one — operators get one polling window of warning state before the auto-abort trips).
**Alternative:** opt-in (operator passes `auto_rollback=true` per rollout). Rejected because the safer default at 2000-host scale is "fail-safe by default, opt out for niche cases." If operators systematically end up overriding, revisit.

### 4. Pending ApplyState at Soak window end

**Decision:** counts as "not applied" — fails the gate if the missing rate pushes apply success below 95%.
**Why:** a host that can't even receive the config push is a real signal, not a transient. If the host can't apply, it's either disconnected or stuck — both should fail the gate.
**Alternative:** treat pending as "unknown, ignore for gate calc." Rejected because it can mask a misconfigured push pipeline (every host getting `pending` and never resolving = silently passing soak with zero applied).

### 5. Promote-phase failure threshold

**Decision:** **no threshold** — Promote completes regardless of failures, surfaces them in ApplyState aggregates.
**Why:** at promote time, the Config has already passed canary + soak. Per-host failures during Promote are usually local-to-the-host issues, not config issues. Auto-aborting on a single Promote failure punishes the entire Scope for one bad host.
**Alternative:** abort if > X% of Promote pushes fail. Reasonable but unvalidated — operator pushback would let us add this in v1.x; for v1.0, "surface failures, let humans decide" is the conservative call.

### 6. Zero-target Canary behavior

**Decision:** abort with `abort_reason=no_canary_targets`.
**Why:** if no hosts are connected when a Phased Rollout starts, there's nothing to canary. Skipping Canary and going straight to Promote (treating Phased like Instant) would be a lie — operator chose Phased for a reason.
**Alternative:** queue the rollout until at least one host connects. Adds a "queued rollouts" state machine; rejected as premature complexity for v1.0.

### 7. Pause clock semantics during Soak

**Decision:** Pause stops the soak clock; Resume re-evaluates the gate immediately, then continues the original soak window from where it was paused.
**Why:** an operator who pauses for 30 minutes while investigating shouldn't be punished by clock reset (full 5min restart) — they paused, they didn't abort.
**Alternative:** clock keeps running during pause (resume = pure no-op). Rejected because "paused for an hour, gate now passes spuriously because the wait window timed out while paused" is a worse failure mode.

### 8. Health regression scope ⚠

**Decision:** **canary subset only**, not full Scope.
**Why:** "any host in the Scope going unhealthy during soak" mixes "this rollout broke things" with "something unrelated happened" — auto-aborting because a non-canary host went down for unrelated reasons is a false positive that erodes trust in the auto-rollback machinery itself.
**Alternative:** full-Scope health regression. Rejected for the false-positive reason above. Could be revisited if operators report missed-detections — but the canary-only signal is what auto-abort is *for*.

### 9. Per-product canary defaults

**Decision:** **not in v1.0** — global defaults only. Per-product overrides land in v1.x.
**Why:** the global default works for the v1.0 operator audience (small ops team, mostly homogeneous fleet). Adding per-product config is real persistence + UI surface that's not load-bearing for v1.0 ship.

### 10. Rollout retention ⚠

**Decision:** **indefinite for v1.0**, mirroring the audit log's append-only retention.
**Why:** at v1.0 fleet size + plausible publish cadence (a few publishes/day per active product), the rollouts table grows slowly. ~10 publishes/day × 365 = ~3,650 rows/year — trivial for either backend.
**Alternative:** time-bounded retention (e.g. 90 days). Reasonable in principle but premature to commit to a number. v1.x can add a `MAGPIE_ROLLOUT_RETENTION_DAYS` knob if storage becomes a concern.

### 11. Audit payload schema versioning

**Decision:** **additive only**; new fields can be added to event payloads, never removed or repurposed.
**Why:** consumers (UI, eventually external audit shippers) can rely on fields being present once they ship. Removing or reusing a field name silently breaks them.
**Alternative:** explicit schema version field in the payload. Defer to v1.x if/when we have a second consumer that needs to negotiate.

### 12. Manual-mode default

**Decision:** **auto**.
**Why:** the operator audience is a small ops team; demanding manual promote-confirmation on every rollout creates friction with no immediate compliance benefit. Manual mode is opt-in for change-management environments that need it.

## Publish dialog ([`v1.0-publish-dialog-spec.md`](./v1.0-publish-dialog-spec.md) §14)

### 1. Confirmation step required at all?

**Decision:** **no separate confirmation modal** after Start Rollout — the publish dialog itself is the confirmation.
**Why:** the dialog already names the blast radius (left pane) and shows the rollout shape (right pane). A second "are you sure?" modal after that much information is theatre.
**Re-evaluate if:** user testing surfaces misclicks. The data we'll have post-launch (audit-log of immediate-aborts after publish) will tell us.

### 2. Default canary inputs

Settled by Rollout #1 above. `5% / N=10`.

### 3. Instant-rollout extra-confirm threshold ⚠

**Decision:** **100 hosts**.
**Why:** below 100, "Instant" is "I'm fixing my dev/staging cluster" territory and the friction of an extra checkbox is annoying. Above 100, "Instant" is "I'm bypassing canary on a real production-scale fleet" and a brake is worth the friction.
**Alternative:** 50, or 500. 100 is order-of-magnitude defensible; the actual threshold is operator-tunable in v1.x if a different number feels right.

### 4. Soak window upper bound

**Decision:** 60 minutes hard limit. Warn-banner on values > 30 min.
**Why:** beyond 30 min, the rollout slot for the Scope is locked for unusually long, blocking concurrent operations. 60 min is a hard wall — operators wanting longer should rethink the rollout shape.

### 5. Tags: free-text or controlled vocabulary?

**Decision:** **free-text** in v1.0.
**Why:** vocabulary requires schema work + operator coordination on what tags mean. Free text is the simplest thing that lets operators record "post-incident-fix-2026-04-12" type context.
**Migration path:** if a vocabulary emerges in practice, v1.x can add an enumeration validator without a breaking change.

### 6. "Last published by" identity field source

**Decision:** **audit-log actor** (the bearer-token-validated identity, not the `X-Magpie-Actor` HTTP header).
**Why:** matches the v0.2 actor semantic — the X-Magpie-Actor label is a hint, never a security claim. Audit-log actor is what's tamper-evident in the hash chain.

### 7. Live progress poll cadence

**Decision:** **2.5s for ApplyState aggregates, 10s for gate evaluation status**.
**Why:** ApplyState updates are cheap (DB aggregate query); polling at 2.5s matches Fleet view cadence. Gate evaluation runs every 10s server-side (Rollout spec §4.3); polling faster than that wastes round-trips.

### 8. Failed-host detail in live progress

**Decision:** **top 20 failed hosts shown inline, rest behind "Show all (148 more)" expand**.
**Why:** at fleet scale, every failed host matters but rendering 148 inline buries the most-recent. 20 is enough for the operator to see the pattern; expand surfaces the rest.

### 9. Operator browser close mid-rollout

**Decision:** **rollout continues server-side**; on reload, the dialog reopens in live-progress view if the rollout is still in flight.
**Why:** rollouts aren't operator-session-bound — closing the tab shouldn't cancel a soak. Reattach is a UX nicety; operators returning to a browser see what's still happening.

### 10. Manual-mode default

Settled by Rollout #12 above. Auto.

### 11. Abort-reason text required?

**Decision:** **optional in v1.0**.
**Why:** demanding text on every abort creates friction. Operators aborting a clearly-failing canary shouldn't have to type "canary failed" — the gate-fail audit event already records why. Optional means operators *can* annotate and it'll be in the `abort_reason` field, but they aren't blocked.

### 12. Drift display when drifting count is large

**Decision:** show count only ("23 hosts drifting"); per-host detail is via Fleet view filter (`?status=drifting`), not embedded in the publish dialog's blast radius pane.
**Why:** keeps the dialog focused on the rollout-being-published; drift forensics belongs in Fleet view.

## Fleet view ([`v1.0-fleet-view-spec.md`](./v1.0-fleet-view-spec.md) §14)

### 1. Aggregate stats endpoint vs `X-Total-Count`

**Decision:** use `X-Total-Count` header (already landed in pagination work).
**Why:** the rollups (healthy / unhealthy / drifting / failed counts) can be derived from the agents API response with status filters. No need for a dedicated `/agents/stats` endpoint when the existing surface is sufficient.
**Re-evaluate if:** at 2000 hosts the per-filter count queries become expensive. v1.x can add a stats endpoint then.

### 2. Cohort dropdown cap behavior

**Decision:** show first 100 entries alphabetically; embed a search-within-dropdown for fleets with > 100 distinct cohorts.
**Why:** at v1.0 scale (one ops team, ~10–30 products) this is unlikely to fire. Cap protects the rare case without forcing UI complexity by default.

### 3. Healthy-host delete warning threshold

**Decision:** "recently healthy" = `last_seen < 5 minutes ago`.
**Why:** matches the stale-`last_seen` amber threshold in the host drawer (consistency). Hosts that pinged in the last 5 minutes are real-live; deleting them is the case the warning catches.

### 4. Sort support per column on `/agents` API ⚠

**Decision:** **client-side sort within loaded page** for v1.0; server-side `?sort=` is v1.x.
**Why:** at the default 500-row page size, client-side sort of the loaded set is fast and doesn't require API changes. The trade-off: sort changes when the operator pages forward (next page sorts within itself, not with the previous page's rows). For v1.0 with bulk operations being the primary multi-page workflow, this is acceptable.
**Re-evaluate if:** operators complain about sort discontinuity across pages. Adding `?sort=last_seen&dir=desc` to the agents handler is mechanical; we just don't want to commit to it before observing the gap.

### 5. Filter+sort state persistence

**Decision:** **session-only** in v1.0 (sessionStorage). Per-user persistence (cookie / server-side) is v1.x.
**Why:** session-only is the simplest persistence that survives accidental tab close + reopen within the same session, which is the common case. Per-user persistence requires identity (RBAC track).

### 6. Search: server-side vs client-side

**Decision:** **client-side within loaded page** for v1.0.
**Why:** server-side `?q=` requires API change; client-side over the loaded 500 rows is instant and good enough for typical "find host db-prod-7" workflows.
**Re-evaluate if:** operators routinely search for hosts beyond the current page (i.e. the loaded page misses).

### 7. Multi-select status filter

**Decision:** **single-select** in v1.0 (status chip is one value).
**Why:** UX simplicity. Operators wanting "unhealthy OR failed" can use two clicks (filter by failed, take action; filter by unhealthy, take action).
**Re-evaluate if:** operators report doing the two-clicks routinely.

### 8. Bulk delete: parallel calls vs `:bulkDelete` endpoint

**Decision:** **N parallel `DELETE /agents/{uid}` calls** for v1.0.
**Why:** no API change required. At up to ~50 deletes per bulk action, the round-trip cost is acceptable and the failure modes are well-isolated (one call's failure doesn't affect the others).
**Re-evaluate if:** the common bulk scale exceeds ~100 deletes (then a `:bulkDelete` endpoint becomes worth the API surface).

### 9. Density toggle (compact / comfortable)

**Decision:** **per-session, defaults to comfortable**.
**Why:** comfortable is more readable for the diagnostic-first workflow; operators who want density can toggle. Per-session matches §5 above.

### 10. "View all 1,840" expanding modal cap

**Decision:** **virtualized list, capped at 5,000 rows**. Above 5k, surface a "save filter and review in chunks" path (deferred to v1.x).
**Why:** 5k virtualized rows are manageable in the browser. Beyond that, scrolling through 1+ million rows in a modal is the wrong UX — operators should narrow their filter first.

### 11. Drift detection lag tooltip

**Decision:** show drift count as-is; if the underlying drift detection is older than 30s (cached), surface a "computed N seconds ago" tooltip on hover.
**Why:** drift can take a beat to detect (the `applied_config_hash ≠ live` comparison happens on poll); the tooltip clarifies if a stale number is on screen.

### 12. `last_seen` rounding in table

**Decision:** **relative time with rounding** ("12s ago", "2m ago", "8d ago"). Absolute timestamp on hover.
**Why:** relative is faster to read at a glance for the most-common operator question ("did this host check in recently?"). Absolute on hover for the operator who needs precision.

## Host drawer ([`v1.0-host-drawer-spec.md`](./v1.0-host-drawer-spec.md) §15)

### 1. Recent activity slice size

**Decision:** **5 events** in drawer; "View all" link goes to Audit section pre-filtered to this host.
**Why:** 5 captures the most-recent meaningful events without scrolling within the drawer. Full history goes to Audit which has its own filter UI.

### 2. YAML diff renderer for "Clear override"

**Decision:** **inline diff (added/removed lines), 50-line truncation** with "View full diff" expander for larger.
**Why:** 50 lines is enough to show the meaningful changes for a typical OTel collector config (which is ~50–200 lines). Full expander handles the outlier case without committing to a heavyweight diff viewer.

### 3. Instance Rollout canary size

**Decision:** **same as product+variant defaults** (canary = 1 = the host); Soak still runs.
**Why:** the resolution priority means an instance Scope has exactly 1 host in it. Canary "= 1" is trivially the whole "fleet"; Soak's purpose is "watch it apply cleanly," which still applies.

### 4. "Open the config running here" target tab

**Decision:** **History tab if the host is drifting; Edit tab otherwise**.
**Why:** drifting → operator probably wants to see what *was* there before they edit (the History tab shows revisions). Non-drifting → operator wants to edit the live config directly.

### 5. Stale `last_seen` thresholds for amber/red

**Decision:** **amber > 5 min, red > 1 hour**.
**Why:** matches the heartbeat cadence operators expect from a live agent. 5 min = "probably fine but watch"; 1 hour = "definitely not heartbeating, investigate."

### 6. Override + label-override coexistence

**Decision:** **both can be active simultaneously**; resolution priority handles the order (instance > product+variant; label override changes the product+variant target).
**Why:** they answer different questions (label = "which cohort?"; config override = "what config?"). Conflating them is the v0.2 trap; spec §10.3 explains.

### 7. Pre-v1.0 audit rows in recent-activity slice

**Decision:** **hidden** because they lack `host_ref`; surface a one-line note in the empty case ("Events from before v1.0 may not be host-tagged").
**Why:** showing untagged events would mix unrelated activity into the host's story. The note prevents operators thinking "no audit history" means "nothing happened."

### 8. Drawer width

**Decision:** **720px desktop default, 100vw mobile, not user-resizable in v1.0**.
**Why:** 720px fits the diagnostic-led layout comfortably. User-resize is a v1.x nicety; not load-bearing.

### 9. Concurrent-edit detection

**Decision:** **none in v1.0** — last-write-wins. Poll cadence (2.5s) catches divergence within the same window.
**Why:** at the v1.0 operator audience size (small ops team), real concurrent-edit conflicts are rare. Adding optimistic-concurrency to every label override / override edit is real work for a problem we don't have yet.

### 10. Multiple in-flight rollouts targeting one host

**Decision:** **render up to 3 cards**; "+N more" link if higher.
**Why:** a host with > 3 in-flight rollouts is an outlier; surfacing them as a count keeps the drawer scannable.

### 11. Pending host display

**Decision:** show Scope resolution chain (server-side resolution is independent of apply state); hide rollouts strip until ApplyState exists; show "not yet reported" for `healthy`.
**Why:** the chain is meaningful even before first apply (it tells the operator what the host *will* get). Empty rollouts strip is invisible (no zero placeholder).

### 12. Affordance order: Re-assign cohort vs Override config

**Decision:** **Re-assign first, Override second**; both visible at once.
**Why:** Re-assign is the more common operation (moving a host between existing cohorts). Override is the less common one (per-host custom config). Order reflects frequency.

## Genuinely consequential decisions (the ⚠ list)

These are the ones to revisit if any feel wrong on first user contact:

1. **Auto-rollback on canary failure default-on** (Rollout §13 #3). Reverses to opt-in if real-world ops show false-aborts.
2. **Health regression scope = canary only** (Rollout §13 #8). Adds full-Scope detection if missed-regression incidents happen.
3. **Rollout retention indefinite** (Rollout §13 #10). Adds `MAGPIE_ROLLOUT_RETENTION_DAYS` if storage growth becomes a concern.
4. **Instant-rollout extra-confirm threshold = 100** (Publish §14 #3). Tune up/down based on the publish cadence above/below 100 hosts.
5. **Sort = client-side within loaded page** (Fleet §14 #4). Adds server-side `?sort=` if cross-page sort discontinuity becomes a complaint.

Everything else in the four spec ⚠ Open Questions sections is a small UX or scoping decision where the working assumption is well-justified and unlikely to need revision.

## What this unblocks

With every working assumption committed, the gates from each spec dissolve:

- **Rollout state machine** — defaults locked, gate semantics locked, retention locked. Code can begin.
- **Publish dialog** — UX defaults locked, threshold logic locked, cadence locked. Code can begin (gated also on Rollout primitive code landing first).
- **Fleet view** — UX defaults locked, sort/search/filter behavior locked. Code can begin (no upstream gate; could land in parallel with Rollout state machine).
- **Host drawer** — UX defaults locked. Code can begin (gated on Fleet view code at minimum, since drawer is row-click target).

Recommended code-implementation order, given the Postgres backend + audit-hash-chain are already wired:

1. **Rollout state machine + ApplyState driver** — pure Go, no UI dependency.
2. **Gate evaluator + auto-abort** — depends on (1).
3. **Operator-action endpoints** (Pause / Resume / Abort / Fast-Promote / Promote) — depends on (1).
4. **Hash-chained audit log Go layer** (compute hash on insert, expose verification endpoint) — independent.
5. **`/metrics` rollout-aware metrics** (`rollout_phase_active{phase}`, `apply_state_total{state}`) — depends on (1).
6. **Fleet view UI** — depends on UI virtualization library choice + cursor-pagination (already landed in API).
7. **Publish dialog UI** — depends on Rollout endpoints (1)+(3).
8. **Rollouts UI section** — depends on Rollout endpoints; the card list view from Fleet view's strip lands here.
9. **Host drawer UI** — depends on Fleet view UI.
10. **Audit query surface UI** — depends on (4).

The (5) audit query surface spec is **still not written** — but the Go-side audit hash chain (item 4 above) is independent and can land any time. The UI for it (item 10) is gated on its spec.
