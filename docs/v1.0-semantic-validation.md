# Magpie v1.0 — Semantic Config Validation

**Status:** Design committed 2026-05-10. Closes plan §6.1 #7 / cut-line item.
**Threat model:** Internal Aptean tool, single trusted ops team (per plan threat model).

---

## 1. Why this exists

Plan §3.2: "Validation runs both structural and semantic — the actual OTel collector config loader runs before the row is accepted. A config that fails semantic validation cannot enter Canary."

Today (v0.2 carryover) Magpie runs structural validation only — YAML well-formedness, pipeline-receiver-exporter references, and the `MAGPIE_ALLOWED_*` allowlist. A YAML that's structurally fine but trips the collector's own loader (missing required field, bad value type, unknown component option) still enters Canary. The canary catches it, aborts the rollout, and the operator pays a cycle for a class of error the system could have refused outright.

Semantic validation closes that gap.

---

## 2. Approach: subprocess `otelcol-contrib validate`

Three approaches were considered:

| Approach | Pros | Cons |
|---|---|---|
| **A. Embed otelcol Go libs** | True validation, no extra binary | Pulls hundreds of MB of indirect deps; locks Magpie to one collector version |
| **B. Subprocess `otelcol-contrib validate`** | Real validation, lean control plane, version-flexible | Requires the binary on the magpied host; subprocess startup latency |
| **C. Hand-rolled rules in Go** | Cheap, no deps | Doesn't catch what the collector's own loader catches; maintenance burden |

**B is the v1.0 choice.** It matches Magpie's existing posture toward upstream otelcol (per agent/CLAUDE.md: "ships otelcol-contrib unmodified, exec's whatever is at `BinaryPath`"). The control plane validates with the same loader the agent eventually uses.

---

## 3. Binary lookup chain

Magpied locates an `otelcol-contrib` binary at startup. First hit wins:

1. `$MAGPIE_OTELCOL_VALIDATOR_PATH` — explicit absolute path. Used in production deploys.
2. `$PATH` lookup for `otelcol-contrib` — convenience for dev / docker-compose where the binary lives next to magpied.

If neither resolves: validator is **disabled with a warning log line**. Rollouts still create; semantic validation is silently skipped.

This is intentional. The plan invariant says "a config that fails semantic validation cannot enter Canary" — *failure* is the gate. A skipped check isn't a failure; it's a degraded state. Hard-failing every publish when the binary is missing during initial v1.0 deploys would be operator-hostile and the warning makes the degraded state visible without breaking the cohort. Operators wanting hard enforcement set `MAGPIE_VALIDATION_REQUIRED=true` (defers to v1.x if not needed sooner).

The publish dialog reflects three states accordingly: `✓ semantic`, `⚠ semantic (skipped — validator binary not configured)`, `✗ semantic` (with stderr).

---

## 4. The validate subprocess

Per call:

1. Write the YAML to a temp file under `os.TempDir()` (clean up on exit).
2. Exec `otelcol-contrib validate --config=<path>` with:
   - 5-second wall-clock timeout (subprocess + collector loader; loader normally completes in tens of ms).
   - stdout/stderr captured into a single string.
3. Exit code 0 → pass.
4. Non-zero or timeout → fail. The captured stderr (truncated to 8 KB) becomes the error detail returned to the publish dialog.

The validator package owns the temp-file lifecycle. Callers see only `Validate(yaml string) error`.

---

## 5. Wiring into the Rollout flow

`internal/rollouts/service.go` Create() runs structural validation after the rollout row is inserted. Semantic validation slots in **right after structural passes**, before any phase transition:

```
structural ✓ → semantic ✓ → enter Canary
structural ✓ → semantic ✗ → markAborted(SemanticValidateFailed) + audit + ErrValidate
```

Same error path as structural failure: rollout enters terminal `aborted` state with `abort_reason="semantic_validate_failed"`, an audit event records the failure detail, and the create call returns `ErrValidate` so the publish dialog renders the inline validate-failed banner.

A new abort reason joins the existing set (`validate_failed`, `no_canary_targets`, `gate_failed`, `operator_aborted`): `semantic_validate_failed`. Distinct so post-incident search can pull just the configs that tripped the loader.

## 6. Service interface

```go
// SemanticValidator validates a collector config YAML using the actual
// OTel collector loader. Implementations are expected to run cheaply
// (≤ 1s typical) but the interface places no upper bound. A nil
// validator means "feature off"; the rollout flow runs structural
// validation only and logs the skip once at startup.
type SemanticValidator interface {
    Validate(ctx context.Context, yaml string) error
}
```

`rollouts.Service` gains an optional `semantic SemanticValidator` field. `NewService` takes it as a parameter; `nil` is permitted and keeps v0.2 behavior. Tests pass `nil` or a stub; production wiring in `cmd/magpied/main.go` resolves the binary lookup chain (§3) and constructs the real implementation.

Concrete impl lives in `internal/semantic/otelcol.go`:

```go
func NewOtelcolValidator(binaryPath string, opts ...Option) *OtelcolValidator
```

Options reserved for future tuning (custom timeout, custom temp dir, extra args). v1.0 first slice ships with sensible defaults baked in.

---

## 7. Latency budget

Subprocess startup on Linux: ~30-50 ms.
Collector loader init: ~50-200 ms (component registry walk).
Total per validate: **~100-300 ms typical, 5 s max**.

Validation latency is observed via the existing `magpie_validation_latency_seconds` histogram (per `ValidationObserver` interface). The histogram now reports the total of structural + semantic; a label split is deferred unless an operator complains.

## 8. Test surface

Unit (no binary required):
- Stub `SemanticValidator` returning canned outcomes — verifies the rollout Create flow honors pass/fail/skip paths and emits the right audit events.
- Service Create with `semantic=nil` — confirms v0.2-compat behavior (no skip warnings, no observer call).

Integration (binary required, build tag `semantic`):
- Real `OtelcolValidator` against a cached otelcol-contrib binary in `internal/semantic/testdata/` (gitignored; downloaded once via `make test-semantic`).
- Round-trip a known-good YAML (passes), a known-bad YAML (fails with stderr containing the loader's error).

CI runs unit only; the integration suite runs locally and on release branches via the OTELCOL_VERSION-aware download script.

---

## 9. Out of scope for this slice

- Auto-extraction of `otelcol-contrib` from a `MAGPIE_RELEASES_DIR` zip. Plan-§6.1 #7 only requires the validator to exist; binary distribution to magpied hosts is an ops concern, not Magpie's.
- Per-platform validators (Windows/macOS magpied hosts). v1.0 magpied runs on Linux containers; the lookup chain still works on other platforms if the binary is present, but we don't ship Windows/macOS test coverage.
- Caching / memoization of validation results across rollouts with identical YAML. Loader latency (~100 ms) is well below the publish-flow budget; caching adds invalidation complexity for marginal gain.
